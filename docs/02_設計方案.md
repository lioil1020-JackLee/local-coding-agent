# 設計方案

## 設計總原則

本專案目前所有設計，應服從以下總原則：

> **以 safe-by-default 為底座，以 CLI-first 為產品表面，以 agent-first 為執行架構，以 skill-enabled 為長期擴充方向。**

這四個原則缺一不可。

---

## 一、Safe-by-default

這是本專案最重要的工程原則。  
任何修改能力若脫離了這個原則，就會直接削弱專案價值。

### 具體意義
- 修改預設在 sandbox/session 中進行
- 不直接污染原始 repo
- 可預覽 diff
- 可在 validation 失敗後中止
- 必要時可 rollback
- 狀態要可追蹤、可回復

### 為什麼重要
若沒有安全底座，agent 再聰明，也只是高風險自動化工具，而不是可信賴的 coding agent。

---

## 二、CLI-first

CLI 不是臨時測試入口，而是產品主介面之一。

### 這代表
- 所有高價值能力都應能由 CLI 觸發
- CLI 輸出要可讀、可腳本化、可整合
- CLI 不應只是直接 expose 低階函式，而應能表達 agent 的高階概念：
  - skills
  - plan
  - run
  - session
  - validate
  - rollback

### 為什麼重要
因為本地端 agent 若沒有穩固 CLI 介面，就很難真正成為獨立產品，也難以與不同 IDE / editor / automation 系統整合。

---

## 三、Agent-first

本專案不是把工具排成一列就叫 agent。  
真正的 agent-first 設計，至少要有：

- 任務輸入
- 規劃
- skill 選擇
- 執行
- 驗證
- 回復
- 狀態追蹤

### 目前實際對應
目前已經初步具備以下 agent loop 雛形：

1. `preview_plan`
2. `select_skill`
3. `execute_skill`
4. `validate_skill`
5. `finalize`

這是很好的方向，但仍屬於第一版骨架，後續還應逐步補上：

- retry policy
- fallback strategy
- richer trace
- multi-step skill chaining
- interactive confirmation

---

## 四、Skill-enabled

### 為什麼需要 skill system
若能力一直散落在工具、服務、pipeline steps 中，系統會越來越難維護，也難以做高層規劃。  
引入 skill system 的目的是把高階能力抽象成可操作單位。

### skill 應具備的基本性質
- 有名稱與描述
- 能決定自己適合什麼任務
- 能產生 plan
- 能執行
- 能驗證
- 能回傳結構化結果

### 目前方向
目前 skill system 已有雛形，例如：

- `analyze_repo`
- `safe_edit`

後續應逐步加入：
- `generate_tests`
- `update_docs`
- `refactor_module`
- `inspect_failure`
- `rollback_changes`

---

## 目前建議的分層架構

### 1. CLI Layer
對使用者暴露命令，例如：
- `skills`
- `plan`
- `run`

未來可擴充：
- `chat`
- `session list`
- `session resume`
- `rollback`
- `diff`

### 2. Agent Runtime Layer
負責高階任務流程控制：
- 理解輸入
- 選 skill
- 跑 plan
- 執行
- 驗證
- 回傳 trace

### 3. Skill Layer
封裝高價值能力。  
這一層是未來擴展性最關鍵的區域。

### 4. Orchestrator / Pipeline Layer
負責既有安全編輯流程的穩定運作，是目前最重要的實作基底之一。

### 5. Service Layer
提供 session、validation、cleanup、status、rollback 等具體能力。

### 6. Session / Sandbox Layer
提供隔離修改與可回復工作區。

---

## 為什麼目前不能貿然推翻既有 pipeline

雖然長期方向是完整 CLI agent，但目前 pipeline 與 orchestrator 仍是穩定主線。  
它們目前承擔：

- safe edit
- validation
- rollback
- session coordination
- test baseline

因此正確方式不是「推翻 pipeline」，而是：

> **在既有穩定 pipeline 之上，建立更高層的 skill 與 agent runtime。**

---

## Session 設計的重要性

未來若要成為真正可用的本地端 agent，session 絕對不能只是暫時資料夾管理，而要視為 agent 持續工作的基礎。

### session 承擔的責任
- 工作區隔離
- 任務狀態保存
- 回復點
- diff 對應
- validation 上下文
- lifecycle 管理
- cleanup 決策依據

---

## 未來設計優先順序

### 第一階段：穩固現有 agent 骨架
- 強化 CLI
- 補 skill registry
- 讓 `run` / `plan` 更一致

### 第二階段：提升 controller / runtime 智能
- retry / fallback
- 更清楚 trace
- 更好的 failure state 管理

### 第三階段：擴充 skill system
- skill manifest
- skill capability tags
- skill routing rules
- skill chaining

### 第四階段：產品化 CLI UX
- 互動式問答
- 確認機制
- diff summary
- session history

---

## 一句話總結

> **本專案的設計方案不是從零做一個理想化 agent，而是在既有 safe-edit pipeline 與 controller 穩定主線上，逐步長出 CLI-first、skill-enabled 的 agent runtime。**
