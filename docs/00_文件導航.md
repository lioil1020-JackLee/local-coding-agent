# 文件導航

## 這份文件的用途

本文件用來幫助新加入的開發者、未來的自己、或任何新的 AI 助理，在最短時間內理解本專案的定位、閱讀順序與各文件的作用。

本專案已明確定義為：

> **本地端 CLI Coding Agent（具備 Skill 系統）**
> 並以逐步達到 **Cursor Agent 等級能力** 為長期目標。

因此，`docs/` 內的文件不只是技術備忘錄，而是整個專案的架構說明、決策脈絡、交接材料與產品方向文件。

---

## 建議閱讀順序

### 第一層：先建立整體心智模型
最先閱讀以下三份文件：

1. `01_目標與方向.md`
2. `02_設計方案.md`
3. `09_變更與進度.md`

這三份文件會回答三個問題：

- 專案最終想成為什麼
- 現在的設計原則是什麼
- 專案目前做到哪裡、下一步在哪裡

---

### 第二層：理解執行主線
若你要碰 execution / orchestrator / controller / pipeline，請繼續讀：

4. `03_pipeline架構.md`
5. `04_ExecutionController說明.md`

這兩份文件會幫你理解：

- 現在的主流程如何運作
- `ExecutionController` 的角色是什麼
- 哪些是穩定基準、哪些是過渡層
- 為什麼現在不能直接把 controller 當成最終 architecture 重寫

---

### 第三層：理解產品介面與協作方式
若你要碰 CLI、Continue、互動流程或團隊協作，再讀：

6. `05_工作流程與Continue.md`
7. `06_合作與貢獻.md`
8. `07_CONTRIBUTING.md`

這一層會幫你釐清：

- CLI 是主產品介面
- Continue 是重要介面層之一，但不是唯一中心
- 如何安全地提出修改
- 如何讓文件與程式一起演進

---

### 第四層：查歷史、查基準、查最近變化
需要看演進歷史與最近狀態時，再讀：

9. `08_CHANGELOG.md`
10. `09_變更與進度.md`

---

## 若你是新助理，最短閱讀路徑

如果你是新的 AI 助理，建議最少照以下順序讀：

1. `01_目標與方向.md`
2. `09_變更與進度.md`
3. `03_pipeline架構.md`
4. `04_ExecutionController說明.md`

讀完後你應該要能回答：

- 這不是單純工具集，而是本地端 CLI agent 專案
- 專案已經有 safe-edit runtime 與 skill system 雛形
- `ExecutionController` 是過渡中的核心邊界
- CLI 與 skill system 是下一階段主軸

## 文件之間的關係

### `01_目標與方向.md`
定義產品最終要去的地方。

### `02_設計方案.md`
說明為了達成那個方向，目前採取的設計原則與分層。

### `03_pipeline架構.md`
說明現有系統如何運作，並把 pipeline 放回 agent runtime 演進脈絡中。

### `04_ExecutionController說明.md`
說明 controller 的現況、責任邊界、相容層角色與未來演進方式。

### `05_工作流程與Continue.md`
說明目前開發與使用方式，以及 Continue 在系統中的定位。

### `06_合作與貢獻.md`
記錄與 AI 長期合作的最佳實踐與交接方式。

### `07_CONTRIBUTING.md`
偏工程規範、開發流程與提交要求。

### `08_CHANGELOG.md`
記錄重大文件與方向更新。

### `09_變更與進度.md`
提供高密度現況摘要。

## 閱讀時應該持有的判斷原則

閱讀本專案任何文件時，請不要只問：

- 目前怎麼跑
- 目前測試過了沒

更要問：

- 這是不是最終架構？
- 這是穩定邊界還是過渡層？
- 這個設計是否讓專案更接近本地端 CLI agent？
- 這個能力未來是否能 skill 化？
- 這是否維持 safe-by-default？
