# 目標與方向

## 專案終極目標

本專案的終極目標已正式明確化為：

> **打造一個本地端 CLI Coding Agent（具備 Skill 系統）**
> 並以逐步達成 **Cursor Agent 等級能力** 為長期產品方向。

這個定義很重要，因為它會直接影響：

- 架構如何分層
- 新功能如何命名
- CLI 如何設計
- skill system 如何抽象
- controller 如何演進
- 文件如何撰寫

---

## 這個專案不再只是什麼

為了避免方向模糊，先講清楚本專案**不再只應被理解為**：

- 一組 Python 工具
- 一個單純的 safe-edit pipeline
- 一組 MCP tools 的封裝層
- 只服務於 Continue 或 IDE 整合的 backend
- 一個只會修改檔案的自動化腳本集合

以上理解都只涵蓋了專案的一部分，但都不足以描述目前的產品方向。

---

## 這個專案應被理解為什麼

本專案應被理解為：

### 1. Local-first
所有核心能力優先在本地端可執行、可測試、可驗證，不依賴雲端專案狀態才能運作。

### 2. CLI-first
CLI 是主要產品表面之一，不是附屬測試入口。  
未來應可直接從命令列完成：

- 任務規劃
- skill 選擇
- 執行
- 驗證
- rollback
- session 管理

### 3. Agent-first
系統核心不是單一 pipeline，而是一個逐步成形的 agent runtime，具備：

- 任務理解
- 規劃
- 執行
- 驗證
- 回復

### 4. Skill-enabled
高價值能力最終都應逐步 skill 化，使系統能：

- 可重用
- 可組合
- 可選擇
- 可擴展

### 5. Safe-by-default
安全不是可選配，而是預設：

- sandbox/session
- diff 預覽
- validation
- rollback
- lifecycle 管理

---

## 為什麼要把方向升級成 CLI Agent + Skill System

如果專案只停在「能安全修改 code 的流程」，會出現幾個長期問題：

1. 能力分散在工具與腳本之間，難以抽象
2. 缺少高層 agent loop，無法做真正 multi-step task
3. CLI 只能當 wrapper，無法成為產品介面
4. 新功能會越做越像堆疊腳本，而不是有中心架構的 agent
5. 與 Cursor Agent 這類產品相比，容易缺少一層“可操作、可組合、可持續使用”的整體體驗

因此，把方向提升為「本地端 CLI agent + skill system」不是改名而已，而是架構與產品心智模型的升級。

---

## 專案最終應具備的能力

### 1. 任務理解能力
能接收自然語言指令，理解使用者要做的事情，並決定適合的 skill 或執行路徑。

### 2. Repo 理解能力
能分析專案結構、檔案分佈、模組關係、重要入口，支援後續規劃與修改。

### 3. Planning 能力
能把高階任務拆成步驟，例如：

- 先分析
- 再規劃修改
- 執行安全修改
- 跑驗證
- 必要時回復

### 4. Execution 能力
能透過 skill 與 pipeline，穩定完成任務執行。

### 5. Validation / Recovery 能力
能在結果不符合預期時，進行：

- 錯誤回報
- fallback
- rollback
- session 復原

### 6. Session 能力
能管理長時任務或跨次互動的狀態，例如：

- create / resume / pin / cleanup
- lifecycle metadata
- current workspace state

### 7. CLI 體驗能力
能從 terminal 直接完成主流程，未來進一步支援更像 Cursor CLI 的互動式體驗。

---

## 現在已經具備哪些基礎

目前專案並不是從零開始，而是已經有一條相當強的基礎主線：

- safe-edit pipeline 已打通
- sandbox / session 模型已存在
- validation / rollback 已存在
- `ExecutionController` 已形成 execution 中樞之一
- CLI 已能列出 skill、預覽 plan、執行 analyze 任務
- 測試已達 `35 passed`

這代表專案已從「概念設計」進入「可執行雛形」階段。

---

## 現在最重要的方向修正

接下來所有新設計與新文件，都應主動對齊以下問題：

1. 這個能力是否服務於 CLI agent？
2. 這個能力未來是否能 skill 化？
3. 這個改動會不會破壞 safe-by-default？
4. 這是否幫助專案朝 Cursor Agent 等級能力收斂？
5. 這是否維持現有 working baseline？

---

## 中期方向

接下來最合理的中期發展順序是：

1. 鞏固現有 CLI 與 controller 主線
2. 補強 skill abstraction / registry / routing
3. 強化真正的 agent loop（plan → select → execute → validate → finalize）
4. 持續收斂 session lifecycle
5. 補上更完整的 CLI UX
6. 最後再深化 Continue / IDE 體驗層

---

## 一句話總結

> **local-coding-agent 的方向，不是停在安全編輯 backend，而是把現有 safe-edit runtime、controller、CLI 與 skills，逐步收斂為一個本地端 CLI Coding Agent。**
