# 03_執行架構與技術主線

## 這份文件在做什麼

這份只講「怎麼執行」。
也就是：收到需求後，系統實際怎麼跑，才會安全又可控。

---

## 單一主線流程

不論 CLI、Chat 或 IDE bridge，最後都走同一條主線：

1. 解析需求（是分析還是修改）
2. 產生 plan（先講清楚要做什麼）
3. 建立 session（修改任務才需要）
4. 執行任務（分析或編輯）
5. 產生 diff
6. 跑 validation
7. 成功就完成；失敗可 rollback
8. 回傳統一 JSON envelope + trace_ref

---

## 任務狀態機

核心狀態固定如下：

1. `planned`
2. `running`
3. `validated`
4. `failed`
5. `rolled_back`

這 5 個狀態是對外契約，避免每個模組各講各話。

---

## Session 生命週期

修改任務的生命週期：

1. create：建立 session 與 sandbox
2. resume：續接既有 session
3. diff：取得本次變更摘要
4. validate：執行驗證流程
5. rollback：必要時還原
6. cleanup：清掉過期資料

---

## 觀測與診斷（trace）

每個任務要記錄最小必要欄位：

1. `task_id`
2. `session_id`
3. `skill`
4. `step`
5. `error`
6. `latency`
7. `checkpoint`

錯誤要能分型：

1. 使用者輸入錯誤
2. 路由錯誤
3. 執行錯誤
4. 驗證錯誤

---

## 對外接口規格

### CLI

所有命令回傳統一 envelope：

1. `ok`
2. `mode`
3. `message`
4. `data`
5. `error`
6. `trace_ref`

### Chat

`/plan /run /status /diff /rollback` payload 與 CLI 對齊。
自然語言任務也必須顯示最終 skill 與 fallback 鏈路。

### IDE bridge（M1）

支援四件事：

1. 送 prompt
2. 取 task status
3. 取 diff/trace
4. 觸發 rollback

另外已補強：

1. 事件流分級（`info/error`）與順序欄位（`seq`）
2. 任務診斷介面（`bridge diagnose`）
3. 佇列診斷統計（`diagnosis_counts`）
4. 診斷輸出可直接附修復命令（`recommended_commands`）
5. 診斷成功時也會附下一步命令（`recommended_next_commands`）
6. 診斷輸出加入白話摘要（`plain_summary`）
7. 診斷輸出加入「你下一句可以直接說」範例（`next_say_examples`）
