# 執行架構與技術主線

## 這份文件的目的

這份文件要說明一件很重要的事：

> **目前的 pipeline 不是終點，也不是舊包袱；它是走向完整 CLI agent 的現役主線。**

如果沒有這個理解，很容易犯兩種錯：

1. 把 pipeline 當成過時設計想全部推翻
2. 把 pipeline 當成最終設計，忽略 skill 與 agent runtime 的上層演進

---

## 現在的 pipeline 承擔什麼角色

目前 pipeline 架構主要負責：

- 建立執行上下文
- 進入 sandbox / session
- 執行修改或分析
- 產生 diff
- 跑 validation
- 回傳結果
- 必要時執行 rollback
- 提供狀態查詢與恢復能力

換句話說，pipeline 目前承擔的是「可靠、安全、可測試」的執行主線。

---

## pipeline 與 agent loop 的關係

不正確理解：

- pipeline = 舊設計
- agent loop = 新設計
- 所以直接刪掉 pipeline

正確理解：

- pipeline = 穩定底層執行主線
- agent loop = 更高層的任務控制與 skill 編排
- skill = 把高階能力模組化後，接到 pipeline 或 service 能力上

合理的架構是：

```text
CLI
  -> Agent Runtime
      -> Skill
          -> Pipeline / Orchestrator
              -> Services
                  -> Session / Sandbox
```

---

## ExecutionController 的定位

`ExecutionController` 是本專案中最容易被誤解、也最容易被過度重構的部分之一。

它目前不是最終形態的 agent runtime，而是：

- 新舊 execution 契約的兼容邊界
- 上層 agent loop 的控制節點
- trace / state / retry 思維的承接點
- skill execution 流程的過渡入口

因此目前不應把它誤認為：

- 已經完成的最終 controller 架構
- 可隨意刪除舊契約的證據
- 足以代表整個 agent runtime 的全部
- 已完全取代 pipeline / orchestrator 的單一真相來源

---

## 目前可觀察到的 agent loop 雛形

目前已出現相對清楚的 execution steps：

1. `preview_plan`
2. `select_skill`
3. `execute_skill`
4. `validate_skill`
5. `finalize`

這代表專案已經從單純的 function call / pipeline entry，進入高階任務流程控制的第一步。

但這還不是完整 agent loop，仍缺少：

- 更細的 retry policy
- failure classification
- fallback skill selection
- interactive confirmation
- richer plan decomposition
- memory / history integration

---

## 技術主線中的關鍵元件

### `run_task_pipeline`
目前仍是重要的穩定入口，未來也可能繼續作為 facade、相容層入口與測試穩定點。

### `EditExecutionOrchestrator`
負責安全修改工作流的重要組織點，是 safe-edit runtime 的主要骨架之一。

### `ExecutionController`
扮演控制邊界與過渡主線角色，不應被誤認為完全取代所有 orchestrator，但確實是更高層控制能力的重要承接點。

---

## Trace Summary 的位置

trace summary 是近一步產品化的重要橋樑，其目標是把 internal execution trace 轉換成 user-facing、可解釋的 CLI 輸出，並建立 single source of truth。

目前的進展重點包括：

- 已建立完整 execution trace pipeline
- trace steps 對應 preview / select / execute / validate / finalize
- 已有 TraceSummaryService
- `trace_summary["text"]`、`trace_summary_text`、`display_message` 已朝單一真相來源收斂
- chat / run / status 已開始整合 trace summary 顯示

這代表系統已從「能執行」往「能解釋自己在做什麼」前進。

---

## 現階段的限制與風險

### pipeline 的限制
- 高層語意還不夠清楚
- 多步任務抽象仍不夠成熟
- 使用者體驗仍偏工程向

### controller 的風險
- 直接理想化重寫，很容易破壞既有 execution contract
- 可能導致 CLI、agent service、session / validation 與測試穩定性回歸

### trace summary 的限制
- output normalization 仍需收斂
- step label 與最終顯示仍需保持一致
- 不應以大重構方式破壞既有測試綠燈

---

## 未來正確的演進順序

1. 維持既有 pipeline、controller 與測試基準的穩定
2. 在上層持續擴充 skill system
3. 讓 CLI 逐步面向 skill 與 agent 概念，而不是底層細節
4. 讓 controller 更像高階 agent runtime，而不是單純轉接層
5. 補更完整的 trace、confirmation、fallback 與 history

---

## 一句話總結

> **目前的 pipeline 是安全執行底座，ExecutionController 是過渡控制邊界，trace summary 是走向可解釋 CLI 體驗的橋樑；三者一起構成現階段最重要的技術主線。**
