# 03_執行架構與技術主線

## 這份文件的目的

本文件要說清楚一件事：

> 目前這個專案的 execution 主線，不是單一 pipeline，也不是單一 controller，而是由 CLI、skill registry、chat/runtime、controller、orchestrator、session / validation / trace services 一起構成的多層執行架構。

如果不把這件事講清楚，就很容易把專案誤寫成過度簡化的版本。

---

## 現在的分層

### 第 1 層：CLI 入口層
由 `repo_guardian_mcp.cli` 提供。

目前可確認的命令包括：

- `skills`
- `plan`
- `run`
- `chat`
- `session list`
- `session resume`
- `diff`
- `rollback`

這層的角色是：

- 接收 repo_root / prompt / task_type / session_id 等輸入
- 建立執行上下文
- 把請求交給 `CLIAgentService` 或 `CLIChatService`
- 輸出結果

---

### 第 2 層：Agent / Chat 協調層
核心是：

- `cli_agent_service.py`
- `cli_chat_service.py`

這一層負責：

- 使用 skill registry
- 分流 plan / run
- 處理 chat turn
- 維護 agent session 狀態
- 管理 working session
- 將自然語言與命令操作接到下層執行

chat mode 目前可確認支援：

- `/help`
- `/skills`
- `/plan`
- `/run`
- `/apply`
- `/status`
- `/session list`
- `/session resume`
- `/diff`
- `/rollback`
- `/exit`

並且若直接輸入自然語言，會優先走 session-aware routing。

---

### 第 3 層：Skill 與 execution control 層
核心包括：

- `skills.py`
- `execution_controller.py`

`skills.py` 提供的不只是兩個 skill 類別名稱，而是整套 skill 基礎結構，包括：

- `SkillContext`
- `SkillPlan`
- `SkillResult`
- `SkillMetadata`
- `SkillManifest`
- manifest 轉 metadata
- generic manifest skill
- skill registry / routing 能力

目前主力 skills 仍是：

- `analyze_repo`
- `safe_edit`

其中 metadata 已包含：

- `aliases`
- `examples`
- `routing_hints`
- `requires_session`
- `requires_validation`
- `priority`
- `enabled`
- `can_chain_to`
- `fallback_skills`
- `manifest_path`

`ExecutionController` 應理解為：

- skill execution 的主要控制節點
- 多層 execution 契約之間的過渡邊界
- orchestration / trace / validation 流程的承接點

---

### 第 4 層：Orchestrator 與 runtime service 層
從 `repo_guardian_mcp/services/` 的命名與目前文件主線可看出，execution 已是多 service 合作的結果，而不是單檔邏輯。

此層的關鍵職責包括：

- 任務執行編排
- edit execution orchestration
- 對話與任務流程協調
- runtime plan / skill graph 支援
- trace summary 彙整

文件在描述此層時，應避免把所有能力都壓縮成單一 pipeline 一詞。

---

### 第 5 層：Session / Sandbox / Validation / Trace 層
這一層構成 safe-by-default 的底座，重點在於：

- session lifecycle
- sandbox 執行隔離
- validation
- diff / rollback
- trace summary

也因為有這一層，專案才不應被理解成高風險的直接改檔腳本。

---

## Pipeline 應如何理解

目前仍可以使用「pipeline」這個詞，但要精準一點。

### 可以這樣寫
目前仍存在可靠的 execution pipeline，用來負責：

- 建立執行上下文
- 進入 session / sandbox
- 執行 skill 或 edit
- 跑 validation
- 產出 diff / rollback / trace 結果

### 不應該這樣寫
- pipeline 就是整個專案
- 只要重寫 pipeline 就等於重寫整個 agent
- pipeline 已經不重要

更精準的說法是：

> pipeline 仍然重要，但它現在是更大 execution architecture 裡的一部分。

---

## `repo_guardian_agent` 在整體架構中的位置

目前 `repo_guardian_agent/` 還在 repo 中，但依 `pyproject.toml` 的 package 與 script 設定，真正的產品主線與對外 CLI 入口在 `repo_guardian_mcp`。

因此較穩妥的寫法是：

> `repo_guardian_agent` 代表 repo 中仍保留的另一組 agent 相關模組；  
> 真正的封裝主線與執行主線在 `repo_guardian_mcp`。

---

## 最後總結

目前最準確的執行架構描述是：

`repo-guardian CLI`
→ `repo_guardian_mcp.cli`
→ `CLIAgentService` / `CLIChatService`
→ `SkillRegistry` / `ExecutionController`
→ orchestrators / runtime services
→ session / sandbox / validation / rollback / trace summary

也就是說，這是一個**已經具備本地 CLI coding agent 骨架，並持續收斂執行主線的專案**。
