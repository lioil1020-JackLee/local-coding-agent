# 執行架構與技術主線

## 這份文件的目的

本文件要說清楚一件事：

> 目前這個專案的 execution 主線，不是單一 pipeline，也不是單一 controller，而是由 CLI、skill registry、chat/runtime、controller、orchestrator、session/validation/trace services 一起構成的多層執行架構。

如果不把這件事講清楚，就會出現兩種錯誤：

1. 把所有東西都誤寫成 pipeline
2. 把 controller 誤寫成已經是最終 agent runtime

這兩種都不精確。

---

## 現在的分層

### 第 1 層：CLI 入口層
由 `repo_guardian_mcp.cli` 提供。

CLI 目前可確認的命令包括：

- `skills`
- `plan`
- `run`
- `chat`
- `session list`
- `session resume`
- `diff`
- `rollback`

這層的角色是：

- 接收 repo_root / prompt / task_type / session_id 等輸入
- 建立 context
- 把請求交給 `CLIAgentService` 或 `CLIChatService`
- 輸出 JSON 結果

---

### 第 2 層：Agent / Chat 協調層
核心是：

- `cli_agent_service.py`
- `cli_chat_service.py`

這一層負責：

- skill registry 的使用
- plan / run 的分流
- chat turn 處理
- agent session 狀態維護
- working session 綁定
- 將自然語言與命令操作接到下層執行

它代表專案已經超出單次命令工具範圍。

---

### 第 3 層：Skill 與 execution control 層
核心包括：

- `skills.py`
- `execution_controller.py`

`skills.py` 的角色不是只放兩個 class 名稱，而是提供：

- `SkillContext`
- `SkillPlan`
- `SkillResult`
- `SkillMetadata`
- `SkillManifest`
- manifest skill 的通用包裝
- skill registry / metadata routing 相關基礎

`ExecutionController` 的角色，應理解為：

- skill execution 的主要控制節點
- 新舊 execution 契約之間的過渡邊界
- orchestration / trace / validation 等流程的承接點

它很重要，但目前不宜被寫成「完整最終 agent runtime」。

---

### 第 4 層：Orchestrator 與 runtime service 層
從 `repo_guardian_mcp/services/` 可見，這層已經相當厚：

- `edit_execution_orchestrator.py`
- `task_orchestrator.py`
- `conversation_orchestrator.py`
- `runtime_plan_service.py`
- `skill_graph_service.py`

這表示 execution 其實是多服務合作的結果，而不是單檔邏輯。

---

### 第 5 層：Session / Sandbox / Validation / Trace 層
主要可見：

- `session_service.py`
- `session_lifecycle_coordinator.py`
- `sandbox_service.py`
- `rollback_service.py`
- `validation_service.py`
- `validation_hook_service.py`
- `trace_summary_service.py`

這一層是 safe-by-default 能否成立的底座。

---

## Pipeline 應如何理解

目前還是可以使用「pipeline」這個詞，但要精準一點。

### 可以這樣寫
目前仍存在可靠的 execution pipeline，用來負責：

- 建立執行上下文
- 進入 session / sandbox
- 執行 skill 或 edit
- 跑 validation
- 產出 diff / rollback / trace 結果

### 不應該這樣寫
- pipeline 就是整個專案
- 只要重寫 pipeline 就等於重寫整個 agent
- pipeline 已經不重要

更精準的說法是：

> pipeline 仍然重要，但它現在是更大 execution architecture 裡的一部分。

## Chat mode 在架構中的位置

chat mode 已不是單純 REPL 外殼。

從 `CLIChatService` 可看出，它已支援：

- `/help`
- `/skills`
- `/plan`
- `/run`
- `/apply`
- `/status`
- `/session list`
- `/session resume`
- `/diff`
- `/rollback`
- `/exit`

並且已經有：

- agent session state
- working session 綁定
- session-aware routing

因此 chat mode 應被視為 agent workflow 的前台介面，而不是單純 demo。

---

## Trace Summary 的位置

`trace_summary_service.py` 已存在於 services 主線中，代表 trace summary 不再只是臨時輸出格式，而是 execution 結果整理的一部分。

文件應將它描述為：

- execution trace 的 user-facing 彙整層
- 讓 CLI 輸出更一致的關鍵服務
- execution flow 的可解釋性接口之一

---

## `repo_guardian_agent` 在整體架構中的位置

目前 `repo_guardian_agent/` 還在 repo 中，但內容規模較小，主要可見：

- `planner.py`
- `patch_generator.py`
- `validation_controller.py`

而 `pyproject.toml` 的 package 與 script 都指向 `repo_guardian_mcp`。  
所以比較穩妥的寫法是：

> `repo_guardian_agent` 目前更像 repo 中保留的另一組模組 / 歷史或輔助路徑；  
> 真正的產品主線與對外 CLI 入口在 `repo_guardian_mcp`。

---

## 最後總結

目前最準確的執行架構描述是：

`repo-guardian CLI`  
→ `cli.py`  
→ `CLIAgentService` / `CLIChatService`  
→ `SkillRegistry` / `ExecutionController`  
→ orchestrators / runtime services  
→ session / sandbox / validation / rollback / trace summary

也就是說，這是一個**已經具備 agent runtime 雛形、但仍在持續收斂主線的本地 CLI coding agent**。
