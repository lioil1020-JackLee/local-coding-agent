# 變更與進度

本文件專門回答三件事：

1. 這個專案到目前為止做了哪些事
2. 現在真正的完成度在哪裡
3. 接下來應優先做什麼

它和 `CHANGELOG.md` 的差別在於：

- `CHANGELOG.md` 偏重里程碑與重大修正
- 本文件偏重「當前狀態總表」與「下一步判斷」

---

## 一、目前真實狀態總結

根據目前可確認的資訊，local-coding-agent 已經不是只有設計文件的專案，而是具備以下實際內容的系統：

- 已有 repo_guardian MCP server 架構
- 已有多個 tools / services
- 已有 session / sandbox / validation / rollback 主線能力
- 已有 ExecutionController 初步實作與測試
- 已與 Continue 設定結合
- 已有 18 項 pytest 測試通過

因此目前最準確的定位是：

> **已完成安全編輯底座與主要工具鏈，正在持續把控制層、規劃層與前端整合做成熟。**

## 二、已完成的部分

### 1. 安全編輯底座
目前最成熟的部分就是安全編輯工作流，包括：
- 建立 session
- 建立 sandbox
- 編輯檔案 / 套用 patch
- 預覽 diff
- 執行 validation
- rollback

這是目前專案最可依賴的能力。

### 2. MCP 工具拆分
相較於早期單一大工具流程，目前已經有明確拆分的小工具思路，這讓整體流程：
- 更容易理解
- 更容易調試
- 更適合 Continue 整合
- 更容易測試

### 3. 測試基準
目前最新已確認基準：
- `pytest`：18 項測試通過

這代表核心流程已具備可運作與可驗證性。

### 4. ExecutionController 初版
ExecutionController 不再只是計畫，而是已有程式與測試痕跡。  
雖未必成熟，但已不是「尚未開始」。

### 5. Continue 規則化
config 已不只是模型設定，而是實際承擔：
- analysis-first
- session workflow 約束
- 專案導覽模式
- 修改任務工具流程限制

## 三、部分完成、但仍需持續強化的部分

### 1. retry / stop / fallback
這部分是目前最重要的半成熟區塊之一。

原因：
- 若沒有明確策略，agent 很容易亂 retry
- 也容易 fallback 到錯誤流程
- 甚至讓工具使用失去可預測性

目前合理判斷是：
- 有控制層基礎
- 但策略仍需文件化與補強

### 2. Planner
目前已有規劃概念與相關 service / 文件描述，但要成為穩定、可用的自然語言規劃器，仍有一段距離。

### 3. Patch Generator / Validation Controller
已有雛形概念與部分工具支撐，但未必已達成熟 agent 拆層程度。

### 4. Continue 體驗收斂
目前已可整合，但「整合」和「穩定好用」是兩件事。  
後者仍需要：
- 更清楚的流程路徑
- 更少 timeout
- 更少歧義
- 更一致的輸出格式

## 四、尚未完成的部分

### 1. 完整多輪記憶

### 2. 更完整 HTTP / API 產品化

但目前實測與主線描述仍應保守地以現有 MCP / stdio 可運作模式為準。

### 3. 高階 refactor 工具鏈
例如：
- rename symbol
- extract function
- 更穩定的 move file orchestration
- 語義級 patch / refactor 能力

### 4. 更完整驗證分級
目前 validation 已有底座，但未來仍需補：
- black
- flake8
- mypy
- pytest 分級

- custom validation rules

## 七、下一步最應優先做什麼

### 第一優先：強化 ExecutionController
原因：
- 它是整個系統從工具集合邁向 agent backend 的核心節點
- 也是 retry / stop / fallback 的承載點
- 若這層不穩，其他高階能力很難穩定落地

### 第二優先：補 validation 分級
原因：
- 安全修改的價值最終要靠 validation 兜底
- 沒有更完整驗證，越高階的 agent 就越危險

### 第三優先：穩定 Continue 體驗
原因：
- 使用者真正感受到的品質，很多時候不是 code 裡的 class，而是互動流程是否穩定

### 第四優先：補成熟規劃與對話能力
原因：
- 沒有多輪記憶與更穩定規劃，代理就很難處理真正複雜任務

5. 下一步優先順序是否改變
