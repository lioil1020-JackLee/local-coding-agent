# 變更與進度

## 這份文件的目的
這份文件是目前專案狀態的高密度交接摘要。未來的新對話如果時間有限，應優先讀這份文件來快速理解：
- 目前做了哪些事
- 過程中踩了哪些坑
- 現在最穩的基準是什麼
- 下一步該怎麼走

## 本輪之前的已知狀態
使用者一開始說明的狀態是：
- `ExecutionController` 已經在做 step orchestration / retry / fallback / trace
- `EditExecutionOrchestrator` 已存在，用於定義 pipeline
- session-based sandbox workflow（copy-based）已穩定
- validation / diff / rollback / session status 已完整打通
- `run_task_pipeline` 是 wrapper
- pipeline 已穩定
- 所有修改都在 sandbox（copy repo）
- session metadata 是 SSOT
- controller 與 tool layer 解耦

當時使用者也明確指出主要風險：
- 每個 session 都 copy repo
- `sandbox_workspaces` 容量暴增
- 需要 session 清理策略與 disk control

## 本輪做的第一階段：架構設計與 skeleton
先完成的不是直接大改，而是先釐清下一階段方向：
- 模組級架構分層
- controller / planner / tools / memory 分層思路
- 實作順序建議
- session cleanup 策略設計（TTL + LRU + pinned）
- code skeleton

## 第二階段：第一次實作與問題
之後開始直接產出可覆蓋的程式碼，包含：
- `ExecutionController`
- `PlanningServiceV2`
- `SessionCleanupService`
- `cleanup_sessions_tool`
- `pin_session_tool`
- 對應測試

這些新增測試可通過，但完整 pytest 爆掉，原因是新 controller 檔案破壞了既有 execution contract。

## 第三階段：相容性修復歷程
一開始採取的方式是逐步補 symbol，例如：
- `ExecutionPlan`
- `FailureKind`
- `ExecutionRequest`
- `ExecutionStatus`

但後來發現真正問題不是少幾個 class name，而是整個 execution contract 同時被兩套代碼依賴。

### 轉折點
使用者一次提供整包專案 zip 後，才改用正確方法：
- 先分析整個 repo 真實契約
- 再一次做完整相容層修正

### 最終結果
`repo_guardian_mcp/services/execution_controller.py` 被修成穩定相容層，並成功讓以下全部恢復正常：
- 新版 handler-based controller tests
- 舊版 `ExecutionRequest / ExecutionPlan / ExecutionStatus` tests
- `run_task_pipeline`
- validation pipeline
- rollback pipeline
- session status pipeline

## 第四階段：session lifecycle phase 1
在 controller 穩住後，又做了 session lifecycle 第一階段接線：
- `create_task_session` 寫入 `last_accessed_at / expires_at / pinned`
- `get_session_status` touch session
- `run_validation_pipeline` touch session
- `rollback_session` touch session
- 新增／正式化：
  - `list_sessions`
  - `resume_session`
  - `pin_session`
  - `cleanup_sessions`

並新增測試 `tests/test_session_lifecycle_tools.py`。

## 第五階段：validation pipeline bug 修復
session lifecycle 接線後，`run_validation_pipeline.py` 曾出現兩次問題：
1. fallback diff 內引用未定義的 `repo_root_path`
2. 修正時又一度產生 broken string literal，造成 SyntaxError

這兩個問題最後都被修好。

## 目前最終狀態
使用者最新本地測試結果：
- `uv run pytest tests/test_validation_pipeline.py -q` → 通過
- `uv run pytest` → **28 passed in 15.82s**

這是目前最重要的狀態結論。
