# 目標與方向

## 專案在做什麼
這個專案的目標不是做一個單純能改檔案的腳本，而是做一個**本地、安全、可控、可回溯的 coding agent backend**。

更具體地說，這個 local-coding-agent 要能幫助使用者：
- 在本地 repo 上進行程式分析
- 在不直接污染原始 repo 的前提下做修改
- 明確看到 diff
- 執行 validation
- 在失敗時 rollback
- 透過 session 機制維持安全與可追蹤性

## 為什麼這個方向重要
很多 agent 工具能做修改，但缺點是：
- 對原始 repo 直接動手
- 缺少 clear review path
- 缺少 rollback
- 缺少 session 隔離
- 一旦多步任務變複雜，狀態很容易亂掉

這個專案的核心價值，就是把這些風險降下來。

## 已經完成的核心基礎
到目前為止，專案已完成或穩定的主線包括：
- `ExecutionController` / pipeline 主線已可穩定運作
- `EditExecutionOrchestrator` 已存在，負責定義 edit pipeline
- session-based sandbox workflow（copy-based）已打通
- validation / diff / rollback / session status 已打通
- `run_task_pipeline` 實際上作為 wrapper 入口存在
- controller 與 tool layer 已具備解耦方向
- session metadata 已被視為 SSOT

## 這輪對話補上的關鍵進展
這輪不是在做「從零開始的設計」，而是在既有穩定基礎上補上兩件很重要的能力：

### 1. ExecutionController compatibility stabilization
`ExecutionController` 現在不是草稿，而是專案當前穩定執行面的核心相容層。這讓新舊兩代 execution contract 可以共存，並保持 pytest 綠燈。

### 2. Session lifecycle phase 1
由於 copy-based sandbox 會造成 workspace 成長，session cleanup 不再只是 TODO，而是已經進入正式實作。

已經落地的方向包括：
- `last_accessed_at`
- `expires_at`
- `pinned`
- `list_sessions`
- `resume_session`
- `pin_session`
- `cleanup_sessions`
- TTL + LRU cleanup 策略

## 目前專案最重要的問題意識
現在真正要小心的不是「功能太少」，而是：
- architecture 正在演進，但不能破壞 working baseline
- session sandbox 會造成磁碟壓力
- execution 主線同時存在新舊契約，必須小心兼容
- 文件若不夠完整，跨對話上下文會流失

## 當前已知穩定基準
本輪合作結束時，使用者本地測試結果為：
- `uv run pytest`
- **28 passed in 15.82s**

這是當前最重要的穩定基準。

## 建議的中期方向
接下來最合理的中期方向不是馬上衝 UI，而是：
1. 讓 session lifecycle 更完整接到 orchestration 主入口
2. 減少 touch/update metadata 邏輯分散
3. 在穩定 controller 之上導入 planner + executor
4. 再做 Continue.dev 深整合
5. 最後才考慮更完整的 memory / UI / API layer

## 對未來新對話的提醒
如果你是新對話的助理，請先理解：
- 這專案不是 blank slate
- safe-edit pipeline 已經是 working system
- `ExecutionController` 目前是穩定相容層
- session lifecycle 已經開始正式落地
- 下一步應該是沿著這個方向收斂，不是重新發散做大改
