# 目標與方向

本文件描述 Local Coding Agent 的整體目標、定位、策略與發展路線，目的不是只列夢想清單，而是要讓未來維護者能直接理解：

- 這個專案真正想解決什麼問題
- 為什麼會選現在這條技術路線
- 目前做到哪裡
- 接下來應該優先做什麼

---

## 一、專案長期目標

本專案的長期目標是打造一個：

> **本地端、可控、安全、可追溯、盡量接近 Cursor 體驗的 Coding Agent**

這個代理系統希望讓使用者能用自然語言完成：

- 分析專案
- 找入口點
- 理解模組關係
- 規劃修改
- 安全修改程式碼
- 預覽差異
- 自動驗證
- 必要時回滾
- 持續進行多步任務

---

## 二、這個專案想解決的核心問題

### 1. 一般 AI 改檔不夠安全
很多 AI coding 流程的問題是：
- 直接改到主線檔案
- 沒有 diff review
- 沒有 validation
- 失敗後難以 rollback
- 對新手不友善

### 2. 不會寫程式的使用者很難安全使用 agent
如果使用者不熟悉：
- Git
- patch
- worktree
- 測試流程
- 架構閱讀

那麼就算 AI 能生成修改，也不代表使用者能安全採用。

### 3. 單一大型工具容易在真實整合情境出問題
早期大工具一口氣做完所有事，看起來方便，但在實際 Continue / MCP 情境中會遇到：
- timeout
- 中途失敗不易定位
- 工具責任不清
- fallback 難以定義
- trace 不易管理

---

## 三、核心方向

### 1. 安全優先，不追求炫技
本專案的首要目標不是炫耀模型能力，而是建立可控流程。

因此專案主線選擇：
- session
- sandbox
- diff
- validation
- rollback

而不是直接讓模型任意改 repo。

### 2. 決定性與可追蹤性
理想狀態下，相同輸入應導向相近或一致的流程結果。  
這也是為什麼需要：
- ExecutionController
- step 結構
- 狀態追蹤
- trace 記錄
- 明確的 retry / stop / fallback 邏輯

### 3. 工具拆分，而不是 mega tool
將流程拆成小工具後，才有機會：
- 更精準觀察問題
- 降低 timeout 風險
- 讓前端更清楚掌握目前在哪一步
- 讓未來 agent controller 真正有控制空間

### 4. 分析與修改分離
專案明確區分：
- **分析任務**：唯讀、理解、整理、搜尋
- **修改任務**：進入 session workflow、安全修改、驗證、回滾

這個分離對新手非常重要，因為它降低了「只是想看懂專案，卻不小心被 AI 改壞檔案」的風險。

---

## 四、目前策略

### 策略 A：以 session workflow 作為寫入主線
所有實際修改都應盡量落在：

```text
preview plan
→ create session
→ edit / stage patch
→ preview diff
→ validation
→ rollback（必要時）
```

### 策略 B：以 ExecutionController 作為控制核心
ExecutionController 的存在意義不是只是多一個 class，而是要逐步成為：

- 步驟控制中心
- 狀態追蹤中心
- retry / stop / fallback 的承載點
- 工具流程的穩定化中樞

### 策略 C：以文件維持長期可接續性
本專案高度依賴文件，因為：
- 使用者不一定會看 code
- AI 也需要文件理解歷史脈絡
- 專案正快速演進，若文件不更新，後續判斷會一直失真

### 策略 D：以 Continue 作為主要互動前端之一
Continue 不只是「能叫到工具」而已，而是：
- 工作流程限制的承載點
- 模型分工的承載點
- 使用者體驗的承載點
- 分析模式與修改模式分離的承載點

## 六、目前最重要的中期目標

### 1. 讓「工具系統」進化成「更完整代理」
目前專案已不只是工具集合，但還需要更成熟的：
- 控制
- 狀態
- 記憶
- 推理流程

### 2. 讓使用者可以更穩定地下模糊自然語言指令
目前已開始有規劃器與規則，但還需要進一步做到：
- 能理解模糊需求
- 能拆解成多步任務
- 能判斷何時只分析、何時能修改
- 能知道何時該停下來而不是亂 fallback

### 3. 讓實作與文件不再脫節
這點非常重要。  
因為目前已經發生過：
- 文件仍寫 14 passed
- 實測其實已是 18 passed
- 文件仍說某些能力未做
- 但程式與測試其實已存在

這會直接影響後續 AI 與人類維護者的判斷，因此必須把文件同步維護視為正式工作，而不是附屬工作。

## 七、未來高優先方向

### A. ExecutionController 強化
需要補強：
- retry guard
- stop guard
- fallback policy
- execution trace 一致性
- no-change / recoverable / fatal error 的狀態語義

### B. Planner 成熟化
需要補強：
- 自然語言拆解
- 任務分類
- 多步計畫輸出
- 和 search / symbol / repo overview 工具的整合

### C. Patch Generator 成熟化
需要補強：
- 多檔案 patch
- 上下文補齊
- 避免插入無關內容
- patch policy
- 對 diff / stage / validation 的對接

### D. Validation 分級
需要從單一驗證進一步擴充為分層驗證：
- syntax / parse
- formatting
- type check
- unit tests
- project command
- custom rules

### E. 多輪記憶與長任務
目前尚未成熟，需要未來補上：
- conversation_id 的穩定追蹤
- session 歷史引用
- 跨輪規劃
- 任務中斷恢復

## 八、長期目標的完成標準（概念）

可以用下面這些問題判斷專案何時接近成熟：

1. 使用者能否用白話要求分析或修改，而不必懂工具名？
2. 修改流程是否總是可追蹤、可驗證、可回滾？
3. Agent 是否能穩定判斷下一步，而不是亂跳流程？
4. 文件是否能真實反映目前狀態？
5. 若一個月後再回來看，是否仍能快速知道：
   - 過去做了什麼
   - 現在做到哪裡
   - 下一步該做什麼

若上述問題大多能答「是」，才表示本專案接近真正的可維護狀態。

---

## 九、總結

本專案的目標不是讓 AI 「看起來很會寫程式」，而是要建立一套：

- 對新手友善
- 對修改安全
- 對流程可控
- 對未來可維護

的本地端 Coding Agent 系統。

也因此，所有未來決策都應持續圍繞這些核心價值：
- 安全
- 控制
- 清楚
- 可驗證
- 可追溯
- 可接續
