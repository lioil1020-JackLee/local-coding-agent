# 目標與方向

## 專案在做什麼
這個專案現在的最高目標，應明確定義為：

> **打造一個本地端 CLI Coding Agent（具備 Skill 系統），並以功能可媲美 Cursor Agent 為終極目標。**

因此，它不應再只被理解為：
- 單純能改檔的腳本
- backend 工具集合
- 一組 MCP tools + pipeline glue

而應被理解為：
- 本地端 agent runtime
- 安全修改系統
- CLI-first 產品
- 未來可 skill 化與組合化的 coding agent

---

## 這個方向為什麼重要
如果只是做「會修改 code 的工具」，很容易出現這些問題：

- 直接污染原始 repo
- 缺少 review path
- 缺少 rollback
- 缺少 session 隔離
- 多步任務狀態容易混亂
- 能力無法模組化重用

而如果目標是「本地端 CLI agent + skill system」，設計思考就會更完整：

- 要有 planning
- 要有 execution control
- 要有 validation / recovery
- 要有 session lifecycle
- 要有 skill abstraction
- 要有可長期使用的 CLI UX

---

## 已完成的核心基礎
到目前為止，專案已完成或穩定的主線包括：

- `ExecutionController` / pipeline 主線可穩定運作
- `EditExecutionOrchestrator` 已存在，定義 edit pipeline
- session-based sandbox workflow（copy-based）已打通
- validation / diff / rollback / session status 已打通
- `run_task_pipeline` 作為 wrapper 入口存在
- controller 與 tool layer 已具備解耦方向
- session metadata 已被視為 SSOT
- session lifecycle phase 1 已接線完成

---

## 現在最重要的定位修正
雖然上述基礎已存在，但文件與心智模型需要統一：

### 不再只是 backend
它是未來 CLI agent 的核心基底。

### 不再只是工具與流程
現有 tools / pipeline steps，未來應逐步 skill 化。

### 不再只是 Continue 整合專案
Continue 是介面層之一，但主體應是本地端 CLI agent runtime。

---

## 目前最重要的問題意識
現在真正要小心的不是「功能太少」，而是：

- architecture 正在演進，但不能破壞 working baseline
- session sandbox 會造成磁碟壓力
- execution 主線同時存在新舊契約，必須小心兼容
- 文件若不夠完整，跨對話上下文會流失
- 未來新增能力若不 skill 化，系統會越來越像工具拼裝而非 agent

---

## 中期方向
最合理的中期方向是：

1. 先讓 session lifecycle 更完整接到 orchestration 主入口
2. 減少 metadata touch/update 邏輯分散
3. 在穩定 controller 之上導入 planner + executor
4. 建立 skill abstraction 與 skill routing
5. 強化 CLI 主流程與 UX
6. 最後再深化 Continue / IDE integration

---

## 最終目標的具體樣子
當專案成熟時，它應該可以被描述為：

- 使用者在本地 CLI 下達任務
- agent 分析 repo 與上下文
- agent 規劃多步修改
- 透過 session sandbox 安全執行
- 自動顯示 diff 與 validation 結果
- 必要時 rollback
- 挑選或組合 skills 完成特定任務
- 整體能力與體驗朝 Cursor Agent 靠攏

---

## 一句話總結
> **這個專案的方向，不是停在 safe-edit backend，而是把 safe-edit runtime 逐步升級為本地端 CLI Agent（具備 Skill 系統），並朝 Cursor Agent 等級能力前進。**
