# 架構總覽與導讀

## 這份文件的用途

這份文件是整合後的入口文件，目的不是重複所有內容，而是讓新加入的開發者、未來的自己，或新的 AI 助理能在最短時間內建立正確心智模型。

本專案已明確定義為：

> **本地端 CLI Coding Agent（具備 Skill 系統）**

因此，`docs/` 內的文件不只是技術備忘錄，而是整個專案的架構說明、決策脈絡、交接材料與產品方向文件。

---

## 建議閱讀順序

### 第一層：先建立整體心智模型
先讀：

1. `02_產品定位與設計原則.md`
2. `04_開發流程_協作與進度.md`

- 專案最終要成為什麼

## 文件之間的關係

### `02_產品定位與設計原則.md`
定義產品最終方向，以及所有新設計應遵守的總原則。

### `03_執行架構與技術主線.md`
說明現有 pipeline、orchestrator、ExecutionController 與 trace summary 在系統中的實際位置。

### `04_開發流程_協作與進度.md`
合併日常開發流程、協作方法、貢獻規範、變更紀錄與目前完成度，是最實用的日常參考文件。

---

## 閱讀時的判斷原則

閱讀本專案任何文件時，不要只問：

- 目前怎麼跑
- 目前測試過了沒

還要問：

- 這是不是最終架構
- 這是穩定邊界還是過渡層
- 這個設計是否讓專案更接近本地端 CLI agent
- 這個能力未來是否能 skill 化
- 這是否維持 safe-by-default

---

## 整合後的文件策略

這次整併的原則是：

- 不再把方向、架構、流程、進度切成過細的小文件
- 保留真正有差異的主題邊界
- 讓 root `README.md` 承擔入口角色
- 讓 `docs/` 只保留少量但高密度的核心文件

一句話總結：

> **本文件負責幫你快速找到路；真正的內容請依序進入產品定位、執行架構與開發進度三條主線。**
