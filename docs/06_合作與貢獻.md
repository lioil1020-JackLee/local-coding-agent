# 合作與貢獻

## 這份文件的目的

本專案不是一次性開發完成，而是透過多輪人機協作逐步演進。  
因此，除了寫程式本身，如何協作、如何交接、如何保留上下文，也是一等重要的設計問題。

---

## 專案的合作現實

這個專案的開發模式，不是傳統多人 PR 流程的簡化版，而更接近：

- 使用者提出目標或問題
- AI 協助分析、設計、產出程式或文件
- 使用者本地覆蓋檔案並執行測試
- 再根據真實結果修正
- 逐步收斂 architecture 與 docs

這種模式的優點是快，但也有風險：

- 容易只修局部而忘了全局
- 容易讓文件落後於實作
- 容易因多輪對話而流失上下文
- 容易在沒有完整 repo context 時做出錯誤假設

---

## 目前最重要的合作原則

### 1. 先保 working baseline
目前本地測試已達 `35 passed`。  
任何重要改動，都應以不破壞這個基準為優先。

### 2. 先看整體，再改局部
尤其是 controller、pipeline、CLI、skills 這些區域，不能只靠 traceback 猜契約。

### 3. 文件與程式要一起更新
這個專案的 docs 不是附件，而是架構交接的一部分。  
若只改 code 不改 docs，後續對話很容易失焦。

### 4. 所有設計都要回到產品方向
每次修改都應回問：
- 這是否讓系統更接近本地端 CLI agent？
- 這是否幫助 skill system 成熟？
- 這是否維持 safe-by-default？

---

## AI 協作時的最佳實踐

### 好的請求方式
- 明確說出目標
- 說出目前錯誤或瓶頸
- 附上本地測試結果
- 指出要改哪些檔案類型
- 若需要，要求整包 zip

### 好的 AI 輸出
- 可直接覆蓋的檔案
- 有結構的修改理由
- 指出風險點
- 提供測試命令
- 同步更新文件

### 不理想的做法
- 只給零碎 patch，沒有全局說明
- 文件極短，導致交接失敗
- 不說明檔案應覆蓋還是新增
- 不保證與現有 baseline 的相容性

---

## 這個專案特別需要重視的協作區域

### 1. `ExecutionController`
因為它牽涉到相容層與 agent loop。

### 2. `skills.py`
因為這會決定 skill system 的未來擴展方式。

### 3. `cli.py`
因為 CLI 是產品表面，命令與輸出結構要穩定。

### 4. `run_task_pipeline`
因為它仍是底層穩定入口之一。

### 5. docs
因為這是跨對話 handoff 的關鍵。

---

## 對未來新助理的協作指引

如果你是未來的新助理，建議這樣做：

1. 先讀 `00_文件導航.md`
2. 再讀 `01_目標與方向.md`
3. 讀 `09_變更與進度.md`
4. 若碰 execution，補讀 `03_pipeline架構.md` 與 `04_ExecutionController說明.md`
5. 若碰 CLI / skill / 產品面，再讀 `02_設計方案.md` 與本檔

然後再開始動程式碼。
