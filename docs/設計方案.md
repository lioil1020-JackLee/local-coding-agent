# 設計方案

## 設計總原則
這個專案的設計方案有一個很清楚的主軸：
**安全修改優先於快速直接修改。**

所以整體設計會偏向：
- 先隔離，再修改
- 先觀察，再提交
- 先驗證，再決定是否保留
- 保持 session metadata 為中心狀態來源

## 目前架構分層理解
雖然專案還在演進，但目前可以這樣理解分層：

### 1. Tool layer
負責暴露使用入口，例如：
- `run_task_pipeline`
- `get_session_status`
- `run_validation_pipeline`
- `rollback_session`
- `list_sessions`
- `resume_session`
- `pin_session`
- `cleanup_sessions`

### 2. Orchestrator / controller layer
負責把多步行為組織起來。
- `task_orchestrator`
- `edit_execution_orchestrator`
- `execution_controller`

其中 `execution_controller` 現階段不是單純新架構，而是相容層。

### 3. Service layer
負責提供具體能力，例如：
- session 管理
- diff
- validation
- rollback
- cleanup

### 4. Session / sandbox runtime
負責實體 sandbox workspace、session metadata 與對應關係。

## 為什麼 session metadata 是 SSOT
session metadata 之所以被視為單一真實來源，是因為它同時承擔：
- session identity
- workspace location
- lifecycle state
- validation / rollback 相關資訊
- edited files / change context

未來不應輕易把 lifecycle 或工作狀態拆散到多個來源，否則很容易讓 rollback、cleanup、resume 出現不一致。

## 為什麼 cleanup 要用 TTL + LRU
這已經在本輪合作中被明確確認。

只用 TTL 的問題：
- 正在反覆查看的 session 也可能被刪掉

只用 LRU 的問題：
- 很舊的垃圾 session 可能一直堆著

因此目前建議策略為：
- `days=3`
- `max_sessions=20`
- pinned 保留
- active/running 保留
- TTL + LRU 併用

## 為什麼先做 lifecycle，再做更大 agent 能力
因為 copy-based sandbox 已經造成實際磁碟壓力。當 session/workspace 持續累積時，這已經不是可忽略的小問題，而是系統健康度問題。

因此目前設計優先序正確應該是：
1. 先讓 lifecycle 可控
2. 再讓 execution 更聰明

## Controller 的設計定位
目前 controller 的最佳定位是：
- execution compatibility boundary
- gradual migration layer
- trace / retry / fallback / state 管理中心

它現在還不是最終 planner-driven architecture 的終點，而是通往那個方向的橋。

## 未來設計方向
在目前基礎上，未來合理的設計方向是：

### A. Planner + Executor
在穩定 controller 上再往上加 planner，讓使用者請求可先被拆成更清楚的執行計畫。

### B. 統一 session lifecycle manager
將目前分散於不同 tool 的 touch/update 邏輯進一步集中。

### C. Continue-aware interaction layer
等 backend 穩定後，再做更好的 IDE/Continue 整合。

### D. Memory / state layer
當 planner 與 lifecycle 穩定後，再考慮 conversation/task/repo memory。

## 對未來新助理的提醒
未來若你要提新設計，請不要跳過目前已存在的 working baseline。好的設計方案應該是：
- 能接在當前主線上
- 能保留測試綠燈
- 能避免破壞 sandbox/session 安全模型
- 能清楚指出 migration path
