# 本地端 Cursor-like Coding Agent 設計方案

本文件整理 local-coding-agent 的整體設計思路、分層架構、工具設計、安全編輯機制、驗證機制、Continue 整合、模型分工，以及未來演進方向。

本文件的角色不是單純畫藍圖，而是要同時回答三件事：

1. 這個系統理論上應該長什麼樣子
2. 目前實作大致落在哪一段
3. 未來應該怎麼延伸，而不破壞現在已成熟的底座

---

## 一、設計原則

### 1. 安全大於方便
任何寫入操作都不應直接對主 repo 造成不可逆影響。  
因此設計上強調：
- session
- sandbox
- diff preview
- validation
- rollback

### 2. 分析與修改分離
分析階段應盡量維持唯讀；  
修改階段才進入安全流程。

### 3. 工具拆分優於大型黑盒流程
工具拆分的目的不是炫技，而是為了：
- 降低 timeout
- 降低複雜度
- 提升 traceability
- 讓控制層有真正的操作空間

### 4. 控制層應獨立
工具本身不應承擔太多隱含決策；  
應由 ExecutionController / Orchestrator 承接流程控制邏輯。

### 5. 對新手友善
設計時必須假設使用者可能：
- 不懂 Git
- 不懂 MCP
- 不懂 patch
- 不懂 Python 專案結構

因此需要：
- 明確流程
- 可讀文件
- 白話輸出
- 降低魔法行為

## 二、高階系統架構

可將整體系統理解為以下分層：

```text
使用者 / UI
↓
Continue / 對話前端
↓
config.yaml（規則、模型、流程限制）
↓
Agent 層（規劃 / 控制 / 補丁 / 驗證協調）
↓
MCP Tool 層
↓
Orchestrator / Controller
↓
Service 層
↓
Sandbox / Session / Validation / Runtime
```

---

## 三、各層職責說明

### 1. 使用者介面層（UI / Continue / 其他前端）
負責：
- 接收自然語言指令
- 顯示分析結果
- 顯示 diff
- 顯示驗證結果
- 提供使用者確認與回報管道

目前主要前端語境為：
- VSCode Continue
- 對話式操作
- 本地人工覆蓋檔案後再測試

### 2. config / 行為規則層
這一層負責約束代理行為，例如：
- 分析任務先分析，不直接改檔
- 模糊專案導覽指令自動走分析模式
- 修改任務優先走 repo_guardian session workflow
- 限制不該使用的路徑或 mega-tool 行為
- 定義不同模型角色與分工

這層實際上是使用體驗的重要組成，而不是單純設定檔。

### 3. Agent 層
理想上應拆成：

#### Planner
- 理解自然語言
- 任務分類
- 產生步驟清單
- 決定先分析還是先修改

#### Patch Generator
- 根據上下文與需求產生 patch proposal
- 控制 patch 是否只包含程式內容
- 避免插入對話文字

#### Validation Controller
- 決定驗證層級
- 執行驗證後判斷是否可持續
- 在失敗時交給 rollback / stop / fallback

#### Conversation / Context Manager
- 管理 conversation_id、session_id、歷史狀態
- 支援多輪任務
- 支援追蹤前一次修改

### 4. MCP Tool 層
負責提供清楚、可調用、單一責任的功能單位。  
工具層應盡量：
- 接口清楚
- 輸入簡潔
- 回傳結構化
- 錯誤可理解
- 避免過度隱藏副作用

### 5. Orchestrator / Controller 層
這層把多個工具與服務串成可理解的流程。

目前最關鍵的是：
- `ExecutionController`
- `run_task_pipeline`
- 各類 task / edit orchestration

這層應逐步承擔：
- 步驟執行順序
- retry / stop / fallback
- trace
- state machine
- no-change / partial-failure / recoverable-failure 判斷

### 6. Service 層
服務層負責真正執行核心邏輯，例如：
- sandbox service
- session service
- validation service
- rollback service
- patch service
- planning service
- report service
- symbol / repo scan / entrypoint service

### 7. Runtime / Sandbox 層
這是實際承受副作用的地方：
- 建立 sandbox
- 寫入檔案
- 產生 diff
- 執行驗證
- 記錄 session metadata
- 支援 rollback

## 四、核心工具設計

根據目前文件、程式結構與狀態整理，核心工具可分成幾類：

### A. 分析類工具
- `repo_overview`
- `search_codebase`
- `get_entrypoints`
- 其他局部分析工具

#### 角色
- 理解 repo
- 搜尋檔案 / symbol / 關鍵字
- 幫助規劃後續修改
- 不產生副作用

---

### B. session / sandbox 類工具
- `create_task_session`
- `get_session_status`

#### 角色
- 建立隔離工作空間
- 管理 session 狀態
- 提供修改流程的基礎上下文

---

### C. 編輯 / patch 類工具
- `edit_file_tool`
- `propose_patch`
- `preview_diff`
- `preview_session_diff`
- `stage_patch`
- `move_file`

#### 角色
- 產生或套用變更
- 預覽差異
- 在 sandbox 內進行實際寫入
- 為高階 refactor 奠基

---

### D. 驗證 / 保護類工具
- `run_validation_pipeline`
- text / semantic guard 類工具
- rollback 類工具

#### 角色
- 檢查修改是否合理
- 阻擋污染輸出
- 在失敗時恢復安全狀態

---

### E. 報告 / 輔助類工具
- report 相關工具
- git status / project command detection 類工具

#### 角色
- 產生摘要
- 輔助診斷
- 補足狀態資訊

## 五、目前主線工作流設計

### 1. 分析任務主線
理想流程：

```text
接收需求
→ 判定為分析任務
→ repo overview / search / entrypoint / code region
→ 回傳白話整理
```

這條線的原則是：
- 不改檔
- 不建立不必要副作用
- 先幫使用者理解，再決定是否要改

### 2. 修改任務主線
目前最重要的主線為：

```text
preview plan
→ create session
→ edit / propose patch / stage patch
→ preview diff
→ run validation
→ rollback（必要）
```

這條線是整個系統目前最成熟的價值核心。

## 六、安全編輯機制

### 1. Session 隔離
每個任務應有獨立 session，避免不同任務彼此污染。

### 2. Sandbox 優先
所有寫入應盡量只發生在 sandbox，而不是直接改動原始 repo。

### 3. Diff 先於套用
即使修改合理，也應先產生 diff 讓使用者與系統都能理解改動內容。

### 4. Validation 後才視為成功
沒有經過 validation 的修改，不應視為已完成。

### 5. Rollback 作為最後保險
任何失敗都應可回復到安全狀態。

## 七、ExecutionController 的設計定位

ExecutionController 是本專案從「工具系統」升級到「代理後端」的重要關鍵。

### 它應該承擔的責任
1. 定義 step 結構
2. 逐步執行流程
3. 管理狀態轉移
4. 記錄 trace
5. 管理 retry
6. 管理 stop
7. 管理 fallback
8. 正規化結果輸出

### 它目前的實際定位
根據現況，更準確的說法是：
- **已存在初步實作**
- **已有測試覆蓋**
- 但尚未保證完整策略全數成熟

因此，未來設計上不應再把它視為「尚未開始」，而應視為：
- 已有地基，需持續強化

## 八、驗證機制設計

理想上，驗證應分層：

### 第 1 層：結構 / 語法
- 檔案是否存在
- patch 是否可套用
- 基本語法是否正確

### 第 2 層：格式 / 靜態檢查
- black
- flake8
- mypy
- 其他靜態規則

### 第 3 層：行為檢查
- pytest
- 專案命令
- 自訂驗證

### 第 4 層：安全規則
- 是否出現聊天污染
- 是否插入不該存在的文字
- 是否違反 patch policy

### 第 5 層：結果語義
- diff 是否為空
- 是否其實沒有改變
- 是否只是重複 append
- 是否形成危險 fallback

目前實作仍在朝這個方向收斂。

## 九、Continue 整合設計

### Continue 的角色
Continue 在本專案中不是單純 UI，而是：
- 使用者自然語言的入口
- 行為規則套用點
- 模型分工落點
- 工具流程策略執行點

### 目前文件需特別注意
過去有些設計文件使用 `.continue/config.json` 的 HTTP 端點例子，這在設計上合理，  
但在目前實際狀態中，應同時記錄：
- 現有 config 以 `config.yaml` 為主
- 已實測 `python -m repo_guardian_mcp.server` 可啟動
- 真正生效的 Continue 設定與本地環境同步方式要明確寫出，不可只留理想範例

### Continue 整合的理想目標
- 使用者不需要知道工具名
- 代理可自動判斷分析 vs 修改
- 修改任務自動走 session workflow
- diff 與驗證結果可清楚回報
- timeout 風險可控

## 十、模型分工設計

理想上可分成以下角色：

### 1. 理解 / 規劃模型
- 理解自然語言需求
- 分解任務
- 產生步驟

### 2. 搜尋 / 補 context 模型
- 幫助找到相關檔案、函式、符號
- 補足 patch 所需上下文

### 3. 補丁生成模型
- 產生 patch proposal
- 控制輸出內容只包含程式相關修改

### 4. 摘要 / 解釋模型
- 將 diff 與驗證結果轉成使用者看得懂的說明

目前 config 已有模型角色分工概念，未來需與 agent 層進一步對齊。

最合理的順序應是：

1. 穩定 session workflow 與主線工具
2. 穩定 ExecutionController
3. 補明確的 retry / stop / fallback policy

7. 再補 conversation memory
8. 最後再做更完整的 HTTP / API 化與高階體驗
