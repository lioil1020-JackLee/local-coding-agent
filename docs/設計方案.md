# 設計方案

## 設計總原則
這個專案的設計總原則，現在應該被完整表述為：

> **安全修改優先於快速直接修改，且整體架構應逐步收斂為本地端 CLI Agent（具備 Skill 系統），並朝 Cursor Agent 等級能力前進。**

所以整體設計會偏向：

- 先隔離，再修改
- 先觀察，再提交
- 先驗證，再保留
- 以 session metadata 為中心狀態來源
- 以 CLI 作為主要產品介面之一
- 以 skill abstraction 作為長期擴充方式

---

## 目前架構分層理解
雖然專案仍在演進，但目前可這樣理解分層：

### 1. Tool layer
負責暴露入口，例如：
- `run_task_pipeline`
- `get_session_status`
- `run_validation_pipeline`
- `rollback_session`
- `list_sessions`
- `resume_session`
- `pin_session`
- `cleanup_sessions`

### 2. Orchestrator / Controller layer
負責組織多步行為：
- `task_orchestrator`
- `edit_execution_orchestrator`
- `execution_controller`

其中 `execution_controller` 目前是相容層。

### 3. Service layer
提供具體能力：
- session 管理
- diff
- validation
- rollback
- cleanup

### 4. Session / Sandbox runtime
負責實體 sandbox workspace、session metadata 與對應關係。

### 5. Future Agent / Skill layer
未來應新增更明確的：
- planner
- executor
- skill registry
- skill routing
- memory / trace abstraction

---

## 為什麼 session metadata 是 SSOT
因為它同時承擔：

- session identity
- workspace location
- lifecycle state
- validation / rollback 相關資訊
- edited files / change context
- 後續 CLI agent 任務恢復所需的狀態

若未來專案要成為長期可用的 CLI agent，這一層更不能分裂。

---

## 為什麼 cleanup 要用 TTL + LRU
目前策略：

- `days=3`
- `max_sessions=20`
- pinned 保留
- active/running 保留
- TTL + LRU 併用

這不是小優化，而是本地 CLI agent 長期運作的必要條件。  
因為 copy-based sandbox 會造成實際磁碟壓力，若 lifecycle 不可控，agent 的可持續使用性會下降。

---

## 為什麼 skill system 是下一個大方向
目前很多能力已經存在，但仍散落在 tools / pipeline / services 之中。  
若要朝 Cursor Agent 等級能力前進，能力必須逐步 skill 化，才能得到：

- 可組合
- 可重用
- 可選擇
- 可延伸
- 可觀察

未來例如可形成：
- repo analysis skill
- refactor skill
- validation skill
- rollback skill
- docs update skill
- test generation skill

---

## Controller 的設計定位
目前 controller 的最佳定位仍是：

- execution compatibility boundary
- gradual migration layer
- trace / retry / fallback / state 管理中心

它現在還不是最終 planner-driven architecture 的終點，而是通往那個方向的橋。

---

## 未來設計方向

### A. Planner + Executor
在穩定 controller 上加 planner，讓請求先形成清楚的執行計畫。

### B. Skill Abstraction
建立 skill 定義、註冊、選擇與執行路由。

### C. CLI Product Surface
把 `run_task_pipeline` 類入口逐步收斂為更完整 CLI 體驗。

### D. Continue-aware Interaction Layer
在 backend 穩定後再深化 IDE/Continue integration。

### E. Memory / State Layer
在 planner 與 lifecycle 穩定後，再考慮 conversation / task / repo memory。

---

## 對未來新助理的提醒
好的新設計應該：

- 接得上當前 working baseline
- 不破壞 sandbox/session 安全模型
- 清楚指出 migration path
- 幫助專案更接近完整 CLI agent + skill system
