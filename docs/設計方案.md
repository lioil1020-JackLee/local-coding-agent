# 本地端 Cursor‑like Coding Agent 設計方案

本文件詳述系統架構、工具設計、編輯與驗證機制、Continue 設定、模型分工、實作順序以及專案目錄結構等，作為長期演進的指引。

## 高階系統架構

1. **使用者介面（UI）**：
   - 使用者以自然語言下指令，可透過 VSCode Continue 外掛、簡易 CLI 或網頁前端。  
   - UI 將指令傳送給代理（Agent），並顯示回傳的結果、diff 預覽與驗證狀態。  

2. **代理（Agent）**：
   - **指令解析與規劃（Planner）**：將白話指令解析成具體可執行的行動，例如「新增單元測試」「修改函式參數」，並產生步驟清單。  
   - **工具選擇與調用（Tool Executor）**：根據步驟呼叫 MCP 工具，如讀檔、寫檔、搜尋程式碼等。  
   - **補丁產生器（Patch Generator）**：依 Planner 建議，透過語言模型生成程式碼修改內容，確保不直接插入聊天文字。  
   - **驗證與回滾控制器（Validation Controller）**：在每次修改後自動執行驗證，若失敗則回滾。  
   - **對話管理**：與使用者互動，提供進度回報與澄清，並管理 conversation_id 與 session_id。  

3. **MCP Server（後端）**：
   - 提供一組 **MCP tools** 供 Agent 呼叫，確保所有操作的副作用可控制。  
   - **工具層（Tool Layer）**：解析輸入、調用服務，並返回結構化結果。  
   - **調度器（Orchestrator Layer）**：將多個服務組合成 pipeline，例如 edit pipeline、search pipeline。  
   - **服務層（Services Layer）**：封裝核心邏輯，如 sandbox 編輯、session 管理、驗證鉤子。  
   - **Sandbox 環境**：每個任務使用 git worktree 建立隔離副本，所有編輯均在 sandbox 進行，不影響主分支。  

4. **資料存放**：
   - `agent_runtime/sessions/<session_id>.json` 保存每個任務的中繼資料（狀態、編輯檔案列表、摘要等）。  
   - `agent_runtime/sandbox_worktrees/<session_id>` 存放每次編輯的 sandbox（git worktree）。  

## MCP 工具設計

採用「單一職責」與「唯讀／有副作用」分離的設計理念。下表列出主要工具（已實作與規劃中）：

| 工具名稱                    | 職責 | 備註 |
|---------------------------|------|------|
| `run_task_pipeline`       | 執行指定 `task_type` 的 pipeline（如 edit、analysis）。| 由 `task_orchestrator` 組合具體服務。 |
| `create_task_session`     | 建立新 session 並建立 sandbox。| 回傳 `session_id` 與相關路徑。 |
| `get_session_status`      | 查詢 session 中繼資料，如狀態、已編輯檔案。| 唯讀工具。 |
| `apply_text_edit`         | 對 sandbox 單一檔案做 append/prepend/replace。| 回傳實際寫入的檔案路徑。 |
| `apply_text_operations`   | 支援多段複合編輯，結構由前端指定。| 回傳所有受影響的檔案。 |
| `preview_session_diff`    | 產生 sandbox 與 repo 原始檔案的 diff。| 若出錯則由 orchestrator fallback。 |
| `update_session_file`     | 更新 session metadata（狀態、驗證結果等）。| 副作用工具。 |
| `run_validation_hook`     | 呼叫靜態檢查、單元測試等驗證邏輯。| 回傳通過/失敗與摘要。 |
| `search_codebase`         | 以關鍵字搜尋程式檔案或函式。| 用於分析階段，唯讀。 |
| `repo_overview`           | 取得專案樹狀結構、主要檔案概要。| 幫助 Agent 建立全局認知。 |
| `propose_patch`           | 呼叫語言模型產生符合策略的補丁建議。| 需傳入指令與相關檔案，返回 patch JSON。 |
| `preview_diff`            | 在記憶體套用 patch 並產生 unified diff。| 不修改實際檔案。 |
| `stage_patch`             | 將 patch 寫入 sandbox。| 產生副作用。 |
| `run_validation_pipeline` | 執行語法檢查、單元測試與自訂規則。| 驗證修改並回傳結果。 |
| `move_file`               | 在 sandbox 中移動或重新命名檔案。| 用於重構操作。 |
| `rename_symbol`           | （規劃中）重新命名符號。| 高階重構工具。 |
| `extract_function`        | （規劃中）抽取程式片段為新函式。| 高階重構工具。 |

所有工具均需使用簡潔明確的輸入參數並回傳結構化資料，錯誤訊息以繁體中文描述。

## 安全編輯機制

1. **Copy‑Based Sandbox**：每個任務在呼叫 `create_task_session` 時建立 sandbox。所有寫入僅影響 sandbox，原始 repo 完全不受影響。  
2. **ExecutionController Pipeline**：使用 `ExecutionController` 包裝每個步驟為 `ExecutionStep`，每步返回 `StepResult`；Controller 統一管理 state、trace、retry、stop 與 fallback。  
3. **Idempotent Design**：所有步驟不得對相同輸入產生不同副作用；若編輯內容相同則 diff 為空，狀態標記為 `no_change`。  
4. **禁止分析階段修改**：分析工具（如 `search_codebase`、`repo_overview`）只能讀取，不產生任何寫入。  
5. **排除對話內容污染程式碼**：Patch Generator 僅插入程式內容，不可寫入聊天文字；必要註解須使用繁體中文且不影響邏輯。  

## 驗證機制

1. **Diff 檢查**：若沒有任何 diff，則拒絕修改並提示「沒有變更」。
2. **驗證鉤子（可插拔）**：
   - **靜態檢查**：使用 `black`、`flake8` 檢查格式，使用 `mypy` 檢查型別。  
   - **單元測試**：透過 `pytest` 執行現有測試，確保功能不被破壞。  
   - **自訂規則**：例如檢查是否引入敏感字詞或違規 API；可在 `run_validation_hook` 中擴充。  
3. **自動回滾**：若驗證失敗，不會呼叫 `persist_session`，並刪除 sandbox；Agent 應提示失敗原因並指導重新生成修補。  

## Continue 設定檔（繼續器配置）

整合 MCP server 與 VSCode Continue 外掛需建立 `.continue/config.json`，其中包含：

- MCP Server 位址，如 `http://localhost:8000`。  
- 可用工具清單，包含 `run_task_pipeline`、`create_task_session`、`propose_patch`、`preview_diff`、`stage_patch`、`run_validation_pipeline`、`move_file`、`rename_symbol`、`extract_function` 等。  
- 互動流程設定：定義使用者提交修改後 MCP 回傳 diff，Continue 會顯示 diff 並允許確認或回滾。  
- 模型設定：可指定不同任務使用不同模型，例如 GPT‑4 用於生成 patch，小模型用於分析摘要。  

## 模型分工

1. **理解與規劃模型**：解析自然語言需求並產生多步驟計劃。  
2. **搜尋與理解模型**：對 repo 進行語意搜尋，找出相關檔案與函式，可利用向量索引。  
3. **補丁生成模型**：依 Planner 指引與搜尋結果生成程式碼修改；需訓練避免插入非程式內容。  
4. **摘要與解釋模型**：將 diff 及驗證結果轉為白話說明回傳使用者。  

代理層應隱藏模型實作細節，透過 API 或本地推理服務調用。  

## 實作順序建議

1. **規劃專案骨架**：建立目錄結構、虛擬環境與基本依賴（uv、fastapi、pytest 等）。  
2. **實作 MCP server skeleton**：使用 FastAPI 建立基本服務，包含 `/run_task_pipeline`、`/tools/<name>` 等路由。  
3. **實作 ExecutionController**：提供步驟管理、retry/stop/fallback 與 trace 記錄。  
4. **實作 Services**：完成 `sandbox_edit_service`、`session_service`、`session_update_service`、`validation_hook_service` 等。  
5. **實作 Orchestrators**：透過 `ExecutionController` 組裝 pipeline，例如 `EditExecutionOrchestrator`、`TaskOrchestrator`，確保步驟合約一致。  
6. **實作 MCP Tools**：將 orchestrators 包裝成可調用的工具函數。  
7. **撰寫單元測試**：驗證 sandbox 建立、編輯、驗證與回滾流程正確。以 pytest 建立 18+ 測試用例作 baseline。  
8. **整合 Agent 層**：實作 Planner、Patch Generator、Validation Controller；透過 HTTP 呼叫 MCP server。  
9. **整合 Continue 外掛**：撰寫 Continue 設定與調用介面，提供 diff 預覽與回滾功能。於 VSCode 測試體驗。  
10. **持續改善與擴充**：新增更多工具（例如重構與檔案搬移），優化模型效果，完善指令解析。  
11. **高階 Refactor 工具**：開發 `move_file`、`rename_symbol`、`extract_function` 等工具，並在 `tool_registry` 註冊。  
12. **進階 Planner 與 Patch Generator**：導入向量搜尋與語言模型生成跨檔案、多步驟的修改計劃與補丁。  
13. **Conversation Memory**：實作 stateful 對話管理，保存 conversation_id、session 狀態與先前計劃，支援多輪交互。  
14. **Server 強化與 CI/CD**：加入認證、日誌、非同步處理及 metrics，並配置 CI 流程（pytest、lint、mypy、black）。

## Python MCP server 範例

以下是一個簡化的 FastAPI MCP server 範例，展示如何註冊工具並呼叫 orchestrator：

```python
# repo_guardian_mcp/server.py
from fastapi import FastAPI, HTTPException
from pydantic import BaseModel
from repo_guardian_mcp.tools.run_task_pipeline import run_task_pipeline

app = FastAPI(title="Local Coding Agent MCP")

class TaskRequest(BaseModel):
    repo_root: str
    relative_path: str = "README.md"
    content: str = ""
    mode: str = "append"
    old_text: str | None = None
    operations: list[dict] | None = None
    task_type: str = "edit"

@app.post("/run_task_pipeline")
async def run_task(request: TaskRequest):
    result = run_task_pipeline(
        repo_root=request.repo_root,
        relative_path=request.relative_path,
        content=request.content,
        mode=request.mode,
        old_text=request.old_text,
        operations=request.operations,
        task_type=request.task_type,
    )
    if not result.get("ok", False):
        raise HTTPException(status_code=400, detail=result.get("error"))
    return result
```

`run_task_pipeline.py` 會從 `task_orchestrator` 建立 pipeline 並返回結果。

## Continue 整合

1. 在 VSCode 安裝 Continue 外掛，設定 MCP server 端點，例如 `http://localhost:8000`。  
2. 建立 `.continue/config.json`，列出各工具的 endpoint、名稱與說明。  
3. 使用者在 VSCode 中選擇要編輯的檔案片段，呼叫 Continue，輸入白話描述即可觸發代理產生計劃並呼叫工具。執行完畢後會顯示 diff 與驗證結果，並可選擇套用或回滾。  

---

本設計方案遵循繁體中文溝通，強調使用者易用性與安全性，將複雜度隱藏在代理與 MCP server 的實作中。透過分層架構與可插拔工具，專案能持續演進，逐步接近 Cursor 的無縫程式編輯體驗。
