# 02_產品定位與設計原則

## 專案定位

本專案目前應被理解為：

> **本地端、CLI-first、具備 skill system、並以 safe edit / session / validation 為底座的 coding agent 專案。**

這個定位比「一組 Python 工具」或「單一 safe-edit pipeline」更貼近現況。

---

## 不應再如何描述這個專案

以下說法都只描述到部分真相：

- 一組單純工具腳本
- 一個只有 MCP tool 的封裝層
- 一條純 pipeline
- 一個只服務於某個 IDE 整合的 backend
- 一個只會直接改檔的自動化腳本集合

原因是：目前代碼已經有 CLI、chat、skill registry、session lifecycle、execution controller、trace summary 與多個 services 協同結構。

---

## 應採用的核心原則

### 1. Safe-by-default
這仍然是最重要的工程底座。

它意味著：

- 不把直接改 repo 當預設路徑
- 重視 session、diff、rollback、validation
- 讓修改行為可追蹤、可中止、可恢復

---

### 2. CLI-first
CLI 不是測試入口而已，而是正式產品介面之一。

目前 CLI 已明確提供：

- `skills`
- `plan`
- `run`
- `chat`
- `session list`
- `session resume`
- `diff`
- `rollback`

因此文件不應把 CLI 描述成附屬工具。

---

### 3. Agent-oriented，而不只是 single-shot pipeline
目前架構已經不只是在處理一次性 edit pipeline。

從 service 與 chat mode 可看出，系統已具備：

- session-aware interaction
- plan 與 run 分流
- working session 綁定
- trace / status / rollback 的持續互動能力

這代表它更接近 agent workflow，而不是單次命令工具。

---

### 4. Skill-enabled
skill system 不是裝飾，而是未來擴充的重要接口。

目前較準確的描述應是：

- 已具備 metadata 基礎欄位
- 已有 alias / examples / routing hints / priority / enabled
- 已有 `can_chain_to` / `fallback_skills`
- 已有 `SkillManifest`
- 已有 `from_dict()` / `from_json_file()`
- 已有 `register_manifest()`
- 仍未形成成熟的完整 plugin ecosystem

也就是說：

> **skill system 已不只是雛形，但也還不適合被誇寫成完整 plugin 化架構。**

---

## 對文件寫法的直接影響

### 應該這樣寫
- 主線 package 是 `repo_guardian_mcp`
- CLI script 是 `repo-guardian`
- `repo_guardian_agent` 是 repo 中仍保留的另一組模組，但不是目前封裝主線
- skill system 正在演進，且已有 manifest 基礎
- execution 是多 service 協作，不應只用單一 pipeline 一詞概括

### 不應該這樣寫
- 把 manifest 寫成不存在
- 把 chat mode 寫成只有最早期幾個指令
- 把某次測試數字寫成永久現況
- 把 `repo_guardian_agent` 寫成目前 package 主體

---

## Continue / 其他整合該如何理解

repo 中有 `continue/` 目錄，但依目前 package 與 CLI 主線來看，專案不應被重新降格描述成「Continue 專案」。

更貼近現況的說法是：

> Continue 或其他整合層，屬於可能的使用面或整合面；  
> 但目前真正的產品主線仍然是本地 CLI agent。

---

## 關於版本術語

文件中最容易混亂的是 skill registry 的版本說法。

目前較穩妥的寫法是：

- 在文件層，寫成「具備 v2 metadata 基礎，並已出現更進一步的 metadata / manifest / chaining / fallback 能力」
- 若要引用 chat help，可說 chat 介面中出現「skill registry v3 metadata」的描述
- 避免把整個專案一次寫死成「現在就是完整 v3」

---

## 結論

本專案的產品定位，最準確的說法不是某個單一功能完成，而是：

> **它已經擁有本地 CLI coding agent 的主要骨架，並正持續把 skill、session、safe edit、trace 與 runtime 組織成更一致的產品。**
