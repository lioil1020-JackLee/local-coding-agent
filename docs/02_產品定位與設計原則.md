# 產品定位與設計原則

## 專案終極目標

本專案的終極目標已明確化為：

> **打造一個本地端 CLI Coding Agent（具備 Skill 系統）**
> 並以逐步達成 **Cursor Agent 等級能力** 為長期產品方向。

這個定義會直接影響：

- 架構如何分層
- 新功能如何命名
- CLI 如何設計
- skill system 如何抽象
- controller 如何演進
- 文件如何撰寫

---

## 這個專案不再只是什麼

本專案不應再只被理解為：

- 一組 Python 工具
- 一條單純的 safe-edit pipeline
- 一組 MCP tools 的封裝層
- 只服務於 Continue 或 IDE 整合的 backend
- 一個只會修改檔案的自動化腳本集合

以上都只描述了局部，不足以代表目前的產品方向。

---

## 這個專案應被理解為什麼

### 1. Local-first
所有核心能力優先在本地端可執行、可測試、可驗證，不依賴雲端狀態才能運作。

### 2. CLI-first
CLI 是主要產品表面之一，不是附屬測試入口。未來應可直接從命令列完成：

- 任務規劃
- skill 選擇
- 執行
- 驗證
- rollback
- session 管理

### 3. Agent-first
系統核心不是單一 pipeline，而是一個逐步成形的 agent runtime，具備：

- 任務理解
- 規劃
- 執行
- 驗證
- 回復
- 狀態追蹤

### 4. Skill-enabled
高價值能力最終都應逐步 skill 化，使系統能：

- 可重用
- 可組合
- 可選擇
- 可擴展

### 5. Safe-by-default
安全不是可選配，而是預設：

- sandbox / session
- diff 預覽
- validation
- rollback
- lifecycle 管理

---

## 設計總原則

本專案目前所有設計，應服從以下總原則：

> **以 safe-by-default 為底座，以 CLI-first 為產品表面，以 agent-first 為執行架構，以 skill-enabled 為長期擴充方向。**

這四個原則缺一不可。

---

## 目前建議的分層架構

### 1. CLI Layer
對使用者暴露命令，例如：
- `skills`
- `plan`
- `run`
- `chat`
- `session list`
- `session resume`
- `rollback`
- `diff`

### 2. Agent Runtime Layer
負責高階任務流程控制：
- 理解輸入
- 選 skill
- 跑 plan
- 執行
- 驗證
- 回傳 trace

### 3. Skill Layer
封裝高價值能力，例如 `analyze_repo`、`safe_edit`，未來可擴充成更多可組合能力。

### 4. Orchestrator / Pipeline Layer
負責既有安全編輯流程的穩定運作，是目前最重要的實作底座之一。

### 5. Service Layer
提供 session、validation、cleanup、status、rollback 等具體能力。

### 6. Session / Sandbox Layer
提供隔離修改與可回復工作區。

---

## 為什麼不能貿然推翻既有 pipeline

雖然長期方向是完整 CLI agent，但目前 pipeline 與 orchestrator 仍是穩定主線。它們承擔：

- safe edit
- validation
- rollback
- session coordination
- test baseline

因此正確方式不是「推翻 pipeline」，而是：

> **在既有穩定 pipeline 之上，建立更高層的 skill 與 agent runtime。**

---

## Session 設計的重要性

未來若要成為真正可用的本地端 agent，session 不能只是暫時資料夾管理，而要視為 agent 持續工作的基礎。

session 承擔的責任包括：

- 工作區隔離
- 任務狀態保存
- 回復點
- diff 對應
- validation 上下文
- lifecycle 管理
- cleanup 決策依據

---

## Continue 的正確定位

Continue 很重要，但它不是本專案的唯一中心。更正確的定位是：

- Continue = 重要互動介面層之一
- CLI = 主產品介面之一
- backend / runtime = 真正能力核心

一個成熟的本地端 coding agent，應該先在 CLI 成形，再自然延伸到 IDE 與其他整合面。

---

## 中期發展順序

接下來最合理的中期順序是：

1. 鞏固現有 CLI 與 controller 主線
2. 補強 skill abstraction / registry / routing
3. 強化真正的 agent loop（plan → select → execute → validate → finalize）
4. 持續收斂 session lifecycle
5. 補上更完整的 CLI UX
6. 再深化 Continue / IDE 體驗層

---

## 一句話總結

> **local-coding-agent 的方向，不是停在安全編輯 backend，而是把現有 safe-edit runtime、controller、CLI 與 skills，逐步收斂為一個本地端 CLI Coding Agent。**
