# 產品定位與設計原則

## 專案定位

本專案目前應被理解為：

> **本地端、CLI-first、具備 skill system、並以 safe edit / session / validation 為底座的 coding agent 專案。**

這個定位比「一組 Python 工具」或「一條 safe-edit pipeline」更貼近現況。

---

## 不應再如何描述這個專案

以下說法都只描述到部分真相：

- 一組單純工具腳本
- 一個只有 MCP tool 的封裝層
- 一條純 pipeline
- 一個只服務於 IDE/Continue 的 backend
- 一個只會直接改檔的自動化腳本集合

原因是：目前程式碼已經有 CLI、chat、skill registry、session lifecycle、execution controller、trace summary 與多個 services 協同結構。

---

## 應採用的核心原則

### 1. Safe-by-default
這仍然是最重要的工程底座。

它意味著：

- 不把「直接改 repo」當預設路徑
- 重視 session、diff、rollback、validation
- 讓修改行為可追蹤、可中止、可恢復

如果沒有這個原則，專案就只是高風險自動化，而不是可信賴的 coding agent。

---

### 2. CLI-first
CLI 不是測試入口而已，而是正式產品介面之一。

從 `repo_guardian_mcp.cli` 可見，CLI 已經明確提供：

- `skills`
- `plan`
- `run`
- `chat`
- `session list`
- `session resume`
- `diff`
- `rollback`

因此產品說明不應把 CLI 描述成附屬工具。

---

### 3. Agent-oriented，而不只是 single-shot pipeline
目前架構已經不只是在處理一次性 edit pipeline。

從 service 與 chat mode 可看出，系統已開始具備：

- session-aware interaction
- plan 與 run 分流
- working session 綁定
- trace / status / rollback 的持續互動能力

這代表它更接近 agent workflow，而不是單次命令工具。

---

### 4. Skill-enabled
skill system 不是裝飾，而是未來擴充的核心接口。

目前較準確的描述應是：

- 已具備 v2 metadata 基礎
- 已有 alias / examples / routing hints / priority / enabled 等欄位
- 已有 manifest model 與 manifest → metadata 的轉換
- 已可註冊 manifest skill
- 仍未完全發展成成熟的 plugin ecosystem

也就是說：**skill system 已不只是雛形，但也還沒到完全 plugin 化。**

---

## 對文件寫法的直接影響

### 應該這樣寫
- 主線 package 是 `repo_guardian_mcp`
- CLI script 是 `repo-guardian`
- `repo_guardian_agent` 是 repo 中仍保留的另一組模組，但不是目前封裝主線
- skill system 正在演進，而不是停在單一固定版本宣告
- execution 是多 service 協作，不應只用單一 pipeline 一詞概括

### 不應該這樣寫
- 直接把某個歷史階段的測試數字當唯一真相
- 把 manifest 寫成「完全不存在」
- 把 chat mode 寫成只有最早期幾個指令
- 把 repo_guardian_agent 寫成目前 package 主體

---

## Continue / IDE 整合應如何理解

`continue/` 目錄存在，但目前從 package 與 CLI 主線來看，專案不應被重新降格描述成「Continue 專案」。

更貼近現況的說法是：

> Continue 或其他 IDE / MCP 整合，屬於可能的使用面或整合面；  
> 但專案目前真正的產品主線仍然是本地 CLI agent。

---

## 關於版本術語

目前文件中最容易混亂的是 skill registry 的版本說法。

較穩妥的寫法：

- **README / docs 層**：寫「v2 metadata 基礎 + 持續往更完整版本演進」
- 避免把整個專案一次寫死成「現在就是完整 v3」

因為從 chat help 字串可看到 `skill registry v3 metadata` 的描述，但整體文件與欄位設計仍帶有明確的 v2 基礎敘事。

---

## 結論

本專案的產品定位，最準確的說法不是「某個單一功能已完成」，而是：

> **它已經擁有本地 CLI coding agent 的主要骨架，並正持續把 skill、session、safe edit、trace 與 runtime 組織成更一致的產品。**
