# 02_產品定位與設計原則

## 這份文件在做什麼

這份只講一件事：
**我們做的是什麼產品、哪些事一定要守住。**

你可以把它當成「不管未來怎麼重構，都不能違反的規則」。

---

## 產品定位（M1 目標）

本專案的 M1 目標不是 demo，而是：

1. 本地端可日常使用的 coding agent
2. 對新手友善，可用繁體中文白話指令
3. 分析與修改分流，流程可追蹤、可回復

一句話：
**先把可靠性做好，再追求花俏互動。**

---

## 一定要守住的 8 條原則

1. 分析預設不改檔
2. 修改一定要有 session
3. 修改後一定要驗證
4. 驗證失敗可回滾
5. 所有關鍵步驟都要留下 trace
6. CLI / Chat / IDE bridge 輸出格式一致
7. 錯誤要可分類、可理解、可重現
8. 文件要和程式同步更新

---

## 系統分層（白話版）

1. 入口層：CLI、Chat、IDE bridge
2. 流程層：狀態機 + 協調器（決定先做什麼）
3. 執行層：編輯、驗證、diff、rollback（真的動手）
4. 可觀測層：trace、health、benchmark、ops
5. 設定層：Continue config、rules、system prompts

分層目的只有一個：
**改 A 不要牽連 B，出錯時能快速定位。**

---

## 你最需要知道的邊界

1. Controller 只做流程決策，不做底層執行
2. Orchestrator 負責把步驟接起來，不吞掉錯誤
3. Service 負責單一職責，輸入輸出穩定
4. Tool 層只是對外接口，不塞商業邏輯

---

## 與 Cursor 體驗對齊的方向

M1 先對齊這些核心體驗：

1. 我用白話下指令，你能理解
2. 你會自己找入口點與影響範圍
3. 你先給計畫，再執行
4. 你改完會自動驗證，失敗能回復
5. 我可以看見過程（trace/diff/status）

---

## 本階段不追求的事

1. 多人同時協作衝突解決
2. 雲端部署與遠端服務依賴
3. 大型插件生態整合

以上不是不要做，是 **M1 不優先**。

---

## 完成判準（M1）

1. 核心 E2E 任務成功率達標（預設 85%）
2. 新增流程測試全綠、既有測試不退步
3. 使用者可用同一條主線完成「分析 -> 修改 -> 驗證 -> 回復」
