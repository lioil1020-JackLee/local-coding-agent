## 這份文件的定位

本文件整合：

- 開發流程

目的不是記錄所有歷史碎片，而是確保文件與代碼不要再脫節。

---

## 目前實際開發流程

目前最合理的工作流是：

1. 在本地 repo 進行設計與修改
2. 用 AI 協助分析、產生草案、整理文件
3. 回到本地 repo 實際覆蓋與驗證
4. 執行 CLI 命令與測試
5. 根據真實結果再更新文件

這代表一個核心原則：

> **本地實跑結果與目前 repo 真實代碼，永遠比舊文件、舊對話與舊測試數字更重要。**

## CLI / chat / session 的文件更新原則

### CLI
當你更新 README 或 docs 時，CLI 能力要以 `repo_guardian_mcp/cli.py` 為準。  
不要只沿用舊版命令清單。

目前應反映的主命令至少包含：

- `skills`
- `plan`
- `run`
- `chat`
- `session list`
- `session resume`
- `diff`
- `rollback`

### chat
chat 指令要以 `repo_guardian_mcp/services/cli_chat_service.py` 為準。

目前文件應能反映：

- `/help`
- `/skills`
- `/plan`
- `/run`
- `/apply`
- `/status`
- `/session list`
- `/session resume`
- `/diff`
- `/rollback`
- `/exit`

如果之後 chat help 再更新，文件也必須跟著修。

## Skill system 的寫法規範

### 目前可以明確寫的
- skill metadata 已有明確基礎欄位
- 已有 `SkillManifest`
- 已有 `from_json_file`
- 已有 manifest → metadata 的轉換
- 已有 generic manifest skill
- 已能描述 chaining / fallback / priority 等 metadata

### 目前不建議寫死的
- 「現在已完全 plugin 化」
- 「現在只有最原始 v2，沒有後續能力」
- 「manifest 還不存在」

目前最安全、也最貼近代碼的說法是：

> **skill system 已有 v2 metadata 與 manifest 基礎，並正往更完整的下一階段演進，但尚未成熟為完整 plugin ecosystem。**

## `repo_guardian_agent` 的文件寫法規範

這一點很重要，因為最容易寫錯。

### 正確寫法
- repo 內仍有 `repo_guardian_agent/`
- 但 `pyproject.toml` 的 package 與 CLI script 主線是 `repo_guardian_mcp`
- 因此 `repo_guardian_agent` 不應再被描述成目前封裝主體

### 不正確寫法
- 「repo_guardian_agent 是目前主要執行入口」
- 「pyproject 已封裝 repo_guardian_agent」
- 「CLI 主線在 repo_guardian_agent」

---

## 測試與進度的寫法規範

### 不要再做的事
- 把某一輪 `pytest` 的 passed 數字永久寫死
- 用舊測試清單判斷目前 repo 狀態
- 沒看 `tests/` 就說某個測試檔不存在

### 應該做的事
- 以 `tests/` 現況為準
- 以本地 `uv run pytest` 為準
- 如果數字會變動，就寫「以本地實測為準」

這樣文件才不會每次變更都立刻過時。

## 文件維護 checklist

每次調整 docs 前，請先確認：

1. `pyproject.toml` 現在封裝哪個 package？
2. CLI script 現在指到哪裡？
3. `repo_guardian_mcp/cli.py` 有哪些命令？
4. `cli_chat_service.py` 現在有哪些 chat 指令？
5. `skills.py` 現在有哪些 metadata / manifest 能力？
6. `tests/` 現在有哪些關鍵測試檔？
7. README 是否仍與 docs 的主線說法一致？

## 目前進度應如何描述

目前最穩妥的進度敘述不是單一數字，而是這段話：

> 專案已具備本地 CLI 主線、chat mode、skill metadata 與 manifest 基礎、execution controller / orchestrator / trace summary 服務，以及 session / diff / rollback / validation 等 agent runtime 基礎能力，並持續朝更一致的本地 coding agent 產品化方向演進。

這種寫法比「某天 35 passed / 40 passed」更不容易過時。

## 協作原則

### 1. 先保主線可用
不要為了理想化重構，破壞 CLI / session / rollback / validation 這些已成形的主線。

### 2. 先看 package 主線，再看歷史目錄
先看 `pyproject.toml` 與 `repo_guardian_mcp/`，再看其他保留模組。

### 3. 文件必須服務真實維護
文件不是裝飾。  
如果 README、docs 和當前代碼不一致，應優先修文件，而不是繼續沿用舊說法。

### 4. 進度文件要講能力，不要只講口號
最好寫「目前具備什麼能力、還缺什麼」，而不是只寫版本名詞。

## 最後結論

從現在開始，文件維護的最核心標準只有一句：

> **以目前 repo 真實代碼與封裝主線為準，而不是以過去某一輪討論的敘述為準。**
