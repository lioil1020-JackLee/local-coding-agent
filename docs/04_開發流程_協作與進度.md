# 開發流程、協作與進度

## 這份文件的定位

本文件合併以下幾類內容：

- 日常開發流程
- Continue 與 CLI 的分工
- 人機協作與 handoff 原則
- 工程層面的貢獻規範
- 變更紀錄與目前完成度

它的目標是提供一份可日常使用、可交接、可更新的單一參考文件。

---

## 目前實際工作流程

本專案目前真實的工作模式，大致如下：

1. 在本地 repo 中進行設計與開發
2. 使用 AI 協助分析、設計、產出程式或文件
3. 將檔案覆蓋到本地 repo
4. 在本地執行測試與 CLI 驗證
5. 根據真實結果繼續修正
6. 最後同步專案與文件

最重要的原則是：

> **本地 repo 的真實測試結果才是最後真相。**

## CLI 與 Continue 的分工

### CLI
CLI 是主產品介面之一，也是最穩定的外部契約。它最適合承擔：

- 自動化
- 測試
- script integration
- session / diff / rollback 操作
- chat / plan / run 等 agent 行為入口

### Continue
Continue 很重要，但應定位為重要互動介面層之一，適合承擔：

- 顯示 plan 與 skill selection
- 顯示 diff / validation
- 顯示 session 狀態
- 承接 human-in-the-loop 互動

結論是：

> **應先讓 CLI 與 runtime 成熟，再把能力向 IDE 可視化延伸。**

---

## 目前最重要的合作原則

### 1. 先保 working baseline
任何重要改動，都應以不破壞現有可運作基準為優先。

### 2. 先看整體，再改局部
尤其是 controller、pipeline、CLI、skills 等區域，不能只靠 traceback 猜契約。

### 3. 文件與程式一起更新
docs 不是附件，而是架構交接的一部分。若只改 code 不改 docs，後續很容易失焦。

### 4. 所有設計都要回到產品方向
每次修改都應回問：
- 這是否讓系統更接近本地端 CLI agent
- 這是否幫助 skill system 成熟
- 這是否維持 safe-by-default

---

## 工程層面的貢獻規範

### 修改前的問題清單
在送出任何重要修改前，請先回答：

1. 這個改動是否讓專案更接近本地端 CLI agent
2. 這個改動是否與 skill system 相容
3. 這個改動是否維持 safe-by-default
4. 這個改動是否破壞 CLI 現有命令行為
5. 這個改動是否破壞現有測試

### 建議的修改流程
1. 先讀整併後文件
2. 再看相關程式與測試
3. 做小範圍、明確目的的修改
4. 執行本地測試
5. 測 CLI 主命令
6. 同步更新文件
