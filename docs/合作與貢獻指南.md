# 合作與貢獻指南

## 這份文件要解決的問題
這個專案不是單人一次性完成，而是使用者與 AI 長期協作、分多輪對話逐步推進。因此這份文件要讓未來的新對話快速理解：
- 我們平常怎麼合作
- 什麼產出形式最有效
- 哪些錯誤方式已經證明不適合這個專案
- 如何讓新一輪協作不丟失上下文

## 我們目前的合作模式
目前最有效的合作方式如下：
1. 使用者描述目前狀態、風險與下一步目標
2. AI 先做架構分析與模組分層建議
3. AI 產出可直接覆蓋本地的檔案
4. 使用者在本地 Windows + PowerShell + uv 環境覆蓋檔案
5. 使用者執行 pytest
6. 使用者把 traceback 或綠燈結果回傳
7. AI 依據真實結果繼續修正
8. 使用者再把本地專案同步到 Google Drive

這代表未來助理不應該只給概念，而要盡量給：
- 可以直接覆蓋的 py 檔
- 必要時整包 zip
- 明確測試指令
- 清楚指出哪些檔案是 overwrite、哪些是新增

## 這次對話學到的重要合作教訓
### 1. 不要只靠局部猜測修 controller
一開始使用者只貼 traceback 時，如果沒有整體檔案內容，AI 可能會一直補 symbol，但真正問題是整體 execution contract drift。這次後來改成使用者一次上傳整包專案 zip，才真正把問題一次解掉。

### 2. 新文件不能太短
使用者明確指出：如果 md 只是精簡摘要，下一個新對話的 AI 會不知道這輪到底做了什麼、做到哪裡、接下來要做什麼。因此 docs 必須能作為跨對話 handoff 材料，而不是只做 release note。

### 3. 本地 pytest 才是最終真相
AI 端可做分析、模擬與推理，但真正可信的是使用者本地 repo 的測試結果。未來任何高風險修改都應以本地 pytest 結果作為最後驗證。

## 貢獻時的基本原則
- 優先保護 working baseline
- 優先保護 safe-edit pipeline
- 優先保護 session metadata 作為 SSOT
- 優先做相容式演進，而非一次性大爆改
- 大改前先確認測試與實際呼叫面

## 適合的提交內容形式
### 好的提交
- 明確針對一個主題，例如 controller 相容層、session lifecycle、validation bug fix
- 搭配對應測試或至少標明如何驗證
- 文件同步更新，讓未來新對話能接手

### 不好的提交
- 很大範圍改動，但沒有說清楚原因與風險
- 只丟概念不給可覆蓋檔案
- 改動與測試沒有對齊
- 文件只寫非常短的摘要，導致上下文流失

## 對未來新助理的協作指引
若你是未來的新助理：
1. 先讀 `docs/文件導航與閱讀順序.md`
2. 再讀 `docs/目標與方向.md`
3. 再讀 `docs/變更與進度.md`
4. 若要碰 execution 主線，再讀：
   - `docs/ExecutionController_v1_實作說明.md`
   - `docs/pipeline_architecture.md`
5. 之後再決定要不要修改 controller / orchestrator / pipeline 相關檔案

## 現在最需要的貢獻方向
以目前專案狀態，最值得持續的方向是：
- session lifecycle 更深整合
- 減少 lifecycle touch logic 分散
- planner + executor 正式分層
- Continue.dev 深整合
