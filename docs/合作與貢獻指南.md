# 合作與貢獻指南

本文件說明 local-coding-agent 的實際協作模式、日常開發流程、貢獻原則與維護注意事項。  
它的目的是讓新加入者、未來的 AI 助理、以及之後回來維護專案的自己，都能快速進入狀況。

---

## 一、先理解：這不是典型的多人 PR 專案

這個專案目前的主要使用情境，不是傳統的：

- feature branch
- pull request
- reviewer comment
- merge queue

而是更接近：

> **AI 與使用者協作式開發**

也就是：
- 使用者用白話描述需求
- AI 協助分析、規劃與修改
- 使用者下載並覆蓋本地檔案
- 使用者執行測試回報結果
- 再進入下一輪

因此本文件會更重視：
- 安全修改
- 文件同步
- 可追蹤性
- 測試基準
- AI 與使用者之間的合作介面

## 三、推薦協作流程

### 情境 A：分析任務
當使用者要求：
- 看懂專案
- 找入口點
- 說明架構
- 找出影響範圍
- 先不要改，只想理解

建議流程：

1. 使用唯讀分析工具
2. 不修改任何檔案
3. 以聊天或文件形式輸出整理結果
4. 若發現文件與 code 不一致，先記錄差異
5. 必要時再進入規劃階段

### 情境 B：修改任務
當使用者要求：
- 幫我改
- 幫我實作
- 套用變更
- 開始修改

建議流程：

1. 先確認需求理解
2. 說明會改哪些檔案與風險
3. 建立 session / sandbox
4. 產生 patch 或進行安全編輯
5. 預覽 diff
6. 執行 validation
7. 驗證失敗時 rollback
8. 更新相關文件
9. 由使用者在本地執行測試確認

## 四、AI 協作模式下的標準操作

### 1. 助理應做的事
- 幫使用者分析 repo
- 幫使用者規劃修改
- 明確告知會改哪些檔案
- 說明風險與驗證方式
- 產出完整檔案或壓縮包
- 必要時補文件，不只補程式碼

### 2. 使用者應做的事
- 覆蓋本地檔案
- 執行測試
- 回報結果
- 若與預期不同，回傳錯誤訊息或 log
- 協助確認實際 Continue / 本地環境是否生效

### 3. 為什麼要這樣分工
因為目前專案使用者不一定會寫程式，但能協助：
- 執行指令
- 貼出結果
- 驗證流程是否正常
- 確認本地與資料來源是否同步

這種合作方式比要求使用者自行 patch / merge / rebase 更實際。

## 五、日常開發流程建議

### 步驟 1：先跑測試
修改前後都建議先執行：

```bash
uv run pytest
```

至少要知道：
- 修改前基準是否健康
- 修改後是否破壞核心流程

### 步驟 2：確認本地設定
尤其與 Continue / config 相關時，要確認：
- 生效的是哪一份 config
- 本地與同步來源是否一致
- MCP server 路徑是否正確

### 步驟 3：以最小必要修改為原則
修改時避免：
- 順手重構太多
- 一次引入多個新概念
- 大規模換架構但沒更新文件
- 混合分析輸出與程式內容

### 步驟 4：同步更新文件
只要涉及下列任一項，就建議同步更新文件：
- workflow
- 工具名稱
- session 流程
- 測試基準
- 架構定位
- roadmap
- Continue 設定方式

### 步驟 5：由使用者做本地驗證
即使 AI 分析與生成看起來合理，也不代表本地一定完全正確。  
最後仍需由使用者在本地確認：
- 測試
- Continue
- server 啟動
- config 生效情況

## 六、貢獻原則

### 原則 1：安全優先於方便
本專案寧可多一層流程，也不要讓修改直接污染主線。

### 原則 2：文件不是附屬品
文件在本專案裡不是「有空再寫」，而是：
- 架構上下文來源
- AI 下次理解專案的依據
- 未來維護的接手點
- 專案狀態對齊工具

### 原則 3：不要靠記憶維護專案
不要假設未來的人或未來的 AI 還記得：
- 為什麼當初改這樣
- 哪條流程才是主線
- 哪些文件已過期

所有關鍵狀態都應明確寫進 md。

### 原則 4：不做無法驗證的修改
若一個改動：
- 無法看 diff
- 無法驗證
- 無法 rollback
- 無法解釋原因

那它就不應該直接進主線文件或主線程式。

## 七、目前最值得貢獻的方向

### 1. ExecutionController 強化
- retry / stop / fallback
- step 狀態語義

- type check
- project command detection
- 自訂規則

### 4. Continue 體驗收斂
- 工具選擇策略
- 分析任務與修改任務分流
- timeout 風險降低
- 更清楚的輸出規則

### 5. 高階工具與多輪能力
- move file
- rename symbol
- extract function
- conversation 記憶
- 長任務追蹤

## 八、提交前的自我檢查清單

在你認為一輪修改完成之前，建議至少檢查：

### 實作面
- [ ] 有沒有改壞既有核心流程？
- [ ] 是否符合 session workflow 主線？
- [ ] 是否避免了直接污染主 repo？
- [ ] 是否保留 diff / validation / rollback 概念？

### 測試面
- [ ] `uv run pytest` 是否仍通過？
- [ ] 若新增新功能，是否應補測試？

### 文件面
- [ ] README 是否仍描述正確？
- [ ] CHANGELOG 是否有同步重大修正？
- [ ] docs/變更與進度 是否有更新？
- [ ] docs/目標與方向 / 設計方案 / 工作流程 是否仍正確？

### 使用者面
- [ ] 不會寫程式的使用者是否仍看得懂？
- [ ] 本次修改是否有清楚說明風險與驗證方式？

## 九、建議的 commit 類型

目前不強制，但推薦維持簡潔一致：

```text
feat: 新功能
fix: 修復 bug
refactor: 重構
docs: 文件更新
test: 測試更新
chore: 雜項整理
```

若未來正式對外或導入更完整版本管理，再考慮擴充格式。

---

## 十、總結

本專案最重要的不是「快速生成很多改動」，而是：

- 讓改動安全
- 讓流程清楚
- 讓測試可驗證
- 讓文件可接續
- 讓新手也能參與

因此最好的貢獻，不一定是寫最多程式，而是讓整個系統：
- 更穩定
- 更可理解
- 更可追蹤
- 更能延續下去
