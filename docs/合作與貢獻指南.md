# 合作與貢獻指南

## 這份文件要解決的問題
這個專案不是一次性完成，而是使用者與 AI 長期協作、分多輪對話逐步推進。  
現在還多了一個更重要的前提：

> 所有協作都應朝 **本地端 CLI Agent（具備 Skill 系統）** 這個終極目標收斂，  
> 而不是停留在局部修 backend 或修工具。

---

## 我們目前的合作模式
目前最有效的合作方式如下：

1. 使用者描述目前狀態、風險與下一步目標
2. AI 先做架構分析與模組分層建議
3. AI 產出可直接覆蓋本地的檔案
4. 使用者在本地 Windows + PowerShell + uv 環境覆蓋檔案
5. 使用者執行 pytest
6. 使用者把 traceback 或綠燈結果回傳
7. AI 依據真實結果繼續修正
8. 使用者再把本地專案同步到 Google Drive

這代表未來助理不應只給概念，而要盡量給：

- 可直接覆蓋的 py / md 檔
- 必要時整包 zip
- 明確測試指令
- 清楚指出 overwrite 與新增檔案

---

## 在新目標下，好的協作應長怎樣
未來每輪協作不只要問「怎麼修這個 bug」，還應該問：

- 這個改動是否讓專案更接近 CLI agent？
- 這個能力是否應該 skill 化？
- 這個設計會不會讓系統更像 Cursor Agent 等級產品？
- 這個改動有沒有破壞 safe-edit baseline？

---

## 這次合作學到的重要教訓

### 1. 不要只靠局部 traceback 猜 execution contract
controller 問題若只補符號，通常治標不治本。  
需要看整包 repo 與測試，才能真正修好相容性。

### 2. 文件不能太短
文件不是 release note，而是跨對話 handoff 材料。  
尤其現在專案方向已提升到 CLI agent + skill system，若只寫摘要，未來很容易失焦。

### 3. 本地 pytest 才是最後真相
AI 可以設計與推理，但真正可信的是使用者本地 repo 的測試結果。

### 4. 要持續維持北極星
每次協作若只修局部，很容易忘記最終產品定位。  
所以未來文件、命名、模組抽象都應盡量朝 CLI agent / skill system 對齊。

---

## 適合的提交形式

### 好的提交
- 圍繞一個主題收斂，例如：
  - controller 相容層
  - session lifecycle
  - skill abstraction
  - CLI UX
- 指出如何驗證
- 同步更新文件，保留跨對話上下文

### 不好的提交
- 大範圍修改但沒說明目的
- 只丟概念，不給可覆蓋檔案
- 不說風險與驗證方式
- 文件過短，導致上下文流失

---

## 對未來新助理的協作指引
若你是未來的新助理，建議順序：

1. 先讀 `docs/文件導航與閱讀順序.md`
2. 再讀 `docs/目標與方向.md`
3. 再讀 `docs/變更與進度.md`
4. 若要碰 execution 主線，再讀：
   - `docs/ExecutionController_v1_實作說明.md`
   - `docs/pipeline_architecture.md`
5. 若要碰產品定位或未來架構，再讀：
   - `docs/設計方案.md`
   - `docs/工作流程與Continue設定.md`
