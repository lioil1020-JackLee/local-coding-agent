# CONTRIBUTING

本文件說明如何參與 local-coding-agent 專案，並特別釐清：  
這個專案的協作模式與一般多人 PR 導向的開源專案並不完全相同。

它目前更接近：

> **AI + 使用者協作式開發專案**

也就是說，實務流程常常是：

- 使用者提出目標
- AI 協助分析與修改
- 使用者在本地覆蓋檔案
- 使用者執行測試
- 再回報結果與進入下一輪

因此，本文件的重點不是 GitHub 社交流程，而是：
- 如何安全參與
- 如何維持文件與實作同步
- 如何避免工具濫用
- 如何讓未來維護者能接續工作

---

## 一、專案目標

本專案希望建立一個本地端的 Cursor-like coding agent，並盡量滿足以下條件：

1. 使用者不需要會寫程式
2. 可以用白話語言描述需求
3. Agent 能分析 repo、找入口點、理解結構
4. Agent 能規劃修改步驟，而不是只會盲改
5. 修改過程安全可控
6. 修改後可自動驗證
7. 必要時可 rollback
8. 長期維護時，文件與實作狀態不脫節

---

## 二、目前協作模式

### 現階段主要模式
目前最常見的流程如下：

1. 使用者提出需求
2. AI 先分析 repo 或需求
3. AI 規劃修改方案
4. AI 產出修改檔案或壓縮包
5. 使用者覆蓋本地檔案
6. 使用者執行測試（例如 `uv run pytest`）
7. 使用者回報結果
8. AI 再做下一輪分析與修改

### 為什麼採這種模式
原因很實際：
- 使用者可能不會寫程式
- 使用者不一定熟悉 Git / branch / cherry-pick / patch review
- 本專案重視本地可控性與安全流程
- 在某些前端工具情境下，直接讓 AI 即時改整個 repo 風險較高

---

## 三、核心開發原則

### 1. 分析與修改必須分離
這是本專案的第一原則。

#### 分析任務時
- 不可修改任何檔案
- 只允許 read-only 操作
- 分析結果應以 chat / 文件形式輸出
- 不可把分析內容直接寫入 repo

#### 修改任務時
- 必須走安全修改流程
- 不可直接把不受控輸出寫入原始 repo
- 必須保留 diff、驗證與 rollback 概念

---

### 2. 所有修改必須走安全 workflow
目前建議主流程為：

```text
preview plan
→ create session
→ edit sandbox / stage patch
→ preview diff
→ validation
→ rollback（必要時）
```

這不是形式要求，而是用來避免以下問題：
- 改壞主線
- 插入無關內容
- 難以追蹤誰改了什麼
- 驗證失敗後無法回復
- 前端工具 timeout 時留下半完成狀態

---

### 3. 不可把聊天內容寫進程式碼
AI 生成程式碼時，嚴禁把以下內容直接寫入 repo：

- 對話說明
- prompt
- 分析報告
- 跟程式邏輯無關的聊天文字
- 示意性說明但非程式內容的句子

這條規則很重要，因為過去在 AI 協作場景中，最容易發生的錯誤之一，就是把對話內容誤當作程式內容寫入檔案。

---

### 4. 註解與說明以繁體中文為主
本專案主要面向的使用者不一定是工程師，因此：

- 新增註解盡量使用繁體中文
- 文件說明一律優先使用繁體中文
- 變更理由應盡量寫清楚
- 註解重點應放在「為什麼這樣做」

---

### 5. 文件必須跟著實作一起更新
這一條在本專案特別重要。

因為本專案高度依賴：
- AI 協作
- 多次對話接續
- 文件作為上下文來源
- 使用者不一定能靠 code 自己判斷進度

因此每次重大變更至少應同步更新：
- `README.md`
- `CHANGELOG.md`
- `docs/變更與進度.md`
- 視需要更新 `docs/目標與方向.md`、`docs/設計方案.md`、`docs/工作流程與Continue設定.md`

---

## 四、目前專案結構（概念）

目前主要結構可以概念化為：

```text
repo_guardian_mcp/
    MCP server、tools、services、控制與執行邏輯

continue/
    Continue 專案內設定模板或規則檔

docs/
    專案設計文件、進度、方向、協作規則

tests/
    pytest 測試

agent_runtime/
    session、sandbox、runtime 相關資料
```

---

## 五、開發時建議遵循的流程

### A. 在修改前
1. 先確認需求是否屬於分析任務還是修改任務
2. 若需求含糊，先做分析與規劃
3. 先找出會影響的檔案
4. 先說明風險與修改範圍
5. 再進入修改流程

### B. 在修改中
1. 優先使用 repo_guardian MCP tools
2. 優先使用 session / sandbox 機制
3. 優先產出可預覽 diff 的結果
4. 避免一次做過大的不必要重構
5. 保持單次修改目標聚焦

### C. 在修改後
1. 執行 `uv run pytest`
2. 若需要，再跑額外驗證工具
3. 更新相關 md 文件
4. 記錄本次改動的原因、影響與後續待辦

## 六、目前測試與品質要求

### 目前已知基準
- pytest：目前已達 **18 項測試通過**

### 最低要求
參與修改時，至少應：
- 不破壞既有測試
- 不讓核心 workflow 失效
- 不讓 README / CHANGELOG / 進度文件與實作嚴重脫節

### 建議額外品質門檻
未來可逐步強化為：
- black
- flake8
- mypy
- 專案自訂 validation rules
- command detection / project-specific validation

---

## 七、貢獻規則

### 1. 維持可追蹤性
每次變更都應能回答：
- 為什麼改
- 改了哪裡
- 風險是什麼
- 如何驗證
- 若失敗怎麼回復

### 2. 不做不必要的大改
本專案目前仍在演進中，因此：
- 不要為了「看起來厲害」就大規模重構
- 不要為了追求完美而重寫成熟的底座
- 不要同時改太多互相耦合的地方卻沒有文件說明

### 3. 尊重主線架構
目前主線是：
- session workflow
- sandbox-based edit
- diff + validation + rollback
- 工具拆分
- 以穩定與可控為優先

若要引入新方向，需清楚說明：
- 為什麼現有方法不足
- 新方案解決什麼問題
- 對 Continue / MCP / 使用者體驗的影響

### 4. 不要只改程式，不改文件
若修改影響下列任一項，就應同步更新文件：
- workflow
- 測試基準
- 工具名稱或角色
- 主要架構定位
- 專案 roadmap
- Continue 使用方式
- 使用者操作步驟

## 八、Commit 建議格式

目前不強制，但建議遵循：

```text
feat: 新功能
fix: 修復 bug
refactor: 重構
docs: 文件更新
test: 測試更新
chore: 雜務調整
```

若未來導入更正式流程，可再擴充成：
- scope
- breaking change
- migration note

## 十、最重要的原則

這個專案的核心理念可以濃縮成一句話：

> **讓不會寫程式的人，也能安全地使用本地 coding agent。**

因此，所有技術選擇最終都應回到這些問題：

- 是否更安全？
- 是否更可控？
- 是否更容易理解？
- 是否更容易驗證？
- 是否更容易回滾？
- 是否更容易讓未來維護者接手？

若答案是否定的，就要重新思考那個改動是否真的值得做。
